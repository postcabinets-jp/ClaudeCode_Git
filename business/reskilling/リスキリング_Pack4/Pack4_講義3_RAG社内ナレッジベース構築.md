## PART 1: RAGとは何か — なぜ社内AIに必要なのか（15分）

### 問題: なぜChatGPTは自社のことを知らないのか

```
あなた: 当社の有給休暇の申請方法を教えてください。
ChatGPT: 一般的な有給休暇の申請方法をお伝えします…
         （※あなたの会社のルールではない一般論が返ってくる）
```

**原因:** LLM（大規模言語モデル）は公開データで学習されている。自社の就業規則、マニュアル、顧客情報は学習データに含まれていない。

### 解決策は3つあるが…

| 方法 | 概要 | コスト | 精度 | 鮮度 |
|------|------|--------|------|------|
| **ファインチューニング** | モデル自体を自社データで再学習 | 非常に高い | 高い | 再学習が必要 |
| **プロンプトに貼り付け** | 毎回関連ドキュメントをコピペ | 低い | 中 | リアルタイム |
| **RAG（検索拡張生成）** | 質問に関連する文書を自動検索→プロンプトに挿入 | 中程度 | 高い | リアルタイム |

→ **RAGが最もバランスが良い。**「プロンプトに貼り付け」の自動化版と考えるとわかりやすい。

### RAGの仕組み（3ステップ）

```
┌─────────────────────────────────────────────────────┐
│                  RAGの処理フロー                      │
│                                                     │
│  ① 質問を受け取る                                    │
│     「有給休暇の申請方法は？」                         │
│              ↓                                      │
│  ② 関連ドキュメントを検索（ベクトル検索）              │
│     → 就業規則.pdf の第3章がヒット                    │
│     → 申請フロー.xlsx の手順部分がヒット               │
│              ↓                                      │
│  ③ 検索結果 + 質問をLLMに渡して回答生成               │
│     「当社の有給休暇申請は、まずSharePointの           │
│      申請フォームから…」                              │
└─────────────────────────────────────────────────────┘
```

### Before / After

**Before（RAGなし）:**
```
社員: 「出張旅費の精算方法を教えて」
ChatGPT: 「一般的に出張旅費は、領収書を経理部門に提出し…」
→ 自社ルールとは違う。結局マニュアルを自分で探す。
```

**After（RAGあり・社内ナレッジ接続済み）:**
```
社員: 「出張旅費の精算方法を教えて」
社内AIアシスタント: 「当社の出張旅費精算は以下の手順です:
  1. 出張後3営業日以内にFreeeの経費精算画面から申請
  2. 領収書はスマホで撮影してアップロード（原本は月末に経理へ）
  3. 上長承認後、翌月25日に給与口座へ振込
  ※ 日当は国内5,000円/海外8,000円（経費規定 第12条）」
→ 自社のルールに基づいた正確な回答。出典も明示。
```

> **💡 RAGの本質**
>
> RAG = **「社内Googleで検索した結果を、AIアシスタントが読んで答えてくれる」**仕組み。
> AIモデル自体を変える必要はなく、**参照するドキュメントを追加・更新するだけ**で知識を最新に保てます。

## PART 2: ベクトルDB入門 — ドキュメントの「埋め込み」を理解する（15分）

### なぜ普通の検索（キーワード検索）ではダメなのか

| 検索方式 | 仕組み | 弱点 |
|---------|--------|------|
| **キーワード検索** | 完全一致・部分一致 | 「有休」で検索しても「年次有給休暇」がヒットしない |
| **ベクトル検索** | 意味の近さで検索 | 「有休」→「年次有給休暇」「休暇申請」もヒット |

### ベクトル（Embedding）とは

テキストを**数百次元の数値ベクトル**に変換すること。意味が近い文章は、ベクトル空間上で近い位置に配置される。

```
「有給休暇の申請」 → [0.12, -0.34, 0.78, ...] ─┐
                                                 │ 近い！
「年次有給の取り方」→ [0.11, -0.32, 0.80, ...] ─┘

「経費精算の方法」 → [0.89, 0.21, -0.45, ...] ← 遠い
```

### ベクトルDB ツール比較

| ツール | 特徴 | おすすめ用途 | 導入の手軽さ |
|--------|------|-------------|-------------|
| **Pinecone** | フルマネージド、高速 | 本番運用、大規模 | 簡単 |
| **Chroma** | 軽量、Python特化 | プロトタイプ、小規模 | 非常に簡単 |
| **Supabase pgvector** | PostgreSQL拡張 | 既存DB活用 | 中程度 |
| **FAISS** | Meta開発、高速検索 | 大規模、ローカル | 中程度 |
| **Weaviate** | スキーマ管理充実 | 構造化データ混在 | 中程度 |

### ドキュメントの取り込みフロー

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 社内文書  │ →  │ チャンク  │ →  │ 埋め込み  │ →  │ベクトルDB │
│ PDF/Word │    │  分割    │    │  変換    │    │  に格納  │
└──────────┘    └──────────┘    └──────────┘    └──────────┘

例: 就業規則.pdf（100ページ）
  → 500文字ずつに分割（200チャンク）
  → 各チャンクをベクトルに変換
  → Chromaに格納
```

### 演習1: チャンク分割を体験しよう（5分）

以下のテキストを「意味のまとまり」で分割してみてください。どこで切るのが良いでしょうか？

```
当社の有給休暇制度について説明します。入社6ヶ月経過後に10日間の有給休暇が
付与されます。その後は1年ごとに1日ずつ加算され、最大20日まで増加します。
有給休暇の申請は、原則として3営業日前までにFreeeの勤怠管理画面から行って
ください。急な体調不良の場合は当日の朝9時までに上長へ電話連絡し、事後申請
が可能です。未消化の有給休暇は翌年度に限り繰り越しが可能です。ただし、
繰越上限は20日とします。なお、有給休暇の計画的付与として、夏季休暇3日間
（8月13日〜15日）と年末年始2日間は全社一斉取得日として設定しています。
```

**分割のポイント:**
- 「制度の概要」「申請方法」「繰越ルール」「計画的付与」で4チャンクに分ける
- 各チャンクに見出しメタデータを付与するとさらに検索精度が上がる

> **💡 チャンク分割の黄金ルール**
>
> - **サイズ**: 200〜500文字が目安（長すぎると関連度が下がる、短すぎると文脈が失われる）
> - **オーバーラップ**: チャンク間で50〜100文字を重複させると切れ目の情報損失を防げる
> - **意味単位**: 可能なら見出し・段落の区切りで分割
> - **メタデータ**: 出典ファイル名・ページ番号・セクション名を付けておく

## PART 3: 社内FAQボットを構築してみよう（15分）

### 最小構成のRAGシステム

```
必要なもの:
1. 社内ドキュメント（PDF / Word / テキスト）
2. 埋め込みモデル（OpenAI text-embedding-3-small 等）
3. ベクトルDB（Chroma / Pinecone）
4. LLM（GPT-4 / Claude）
5. Python実行環境

→ 50行のPythonコードで動く！
```

### 実装コード（Chroma + OpenAI）

**Step 1: セットアップ**
```bash
pip install chromadb openai langchain langchain-openai
```

**Step 2: ドキュメントの読み込みと格納**
```python
from langchain_openai import OpenAIEmbeddings
from langchain_community.vectorstores import Chroma
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_community.document_loaders import DirectoryLoader, TextLoader

# 1. ドキュメント読み込み（docs/ フォルダ内のテキストファイル）
loader = DirectoryLoader("docs/", glob="**/*.txt", loader_cls=TextLoader)
documents = loader.load()

# 2. チャンク分割
splitter = RecursiveCharacterTextSplitter(
    chunk_size=400,        # 400文字ずつ
    chunk_overlap=50,      # 50文字オーバーラップ
    separators=["\n\n", "\n", "。", "、", " "]
)
chunks = splitter.split_documents(documents)
print(f"チャンク数: {len(chunks)}")

# 3. ベクトルDBに格納
vectordb = Chroma.from_documents(
    documents=chunks,
    embedding=OpenAIEmbeddings(model="text-embedding-3-small"),
    persist_directory="./chroma_db"
)
print("ベクトルDB構築完了！")
```

**Step 3: 質問応答**
```python
from langchain_openai import ChatOpenAI
from langchain.chains import RetrievalQA

# 4. RAGチェーンの構築
llm = ChatOpenAI(model="gpt-4o", temperature=0)
qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    chain_type="stuff",
    retriever=vectordb.as_retriever(search_kwargs={"k": 3}),
    return_source_documents=True
)

# 5. 質問してみる
result = qa_chain.invoke({"query": "有給休暇の申請方法を教えてください"})
print("回答:", result["result"])
print("参照元:", [doc.metadata for doc in result["source_documents"]])
```

**出力イメージ:**
```
回答: 有給休暇の申請は、原則として3営業日前までに
Freeeの勤怠管理画面から行ってください。急な体調不良の
場合は当日朝9時までに上長へ電話連絡し、事後申請が可能です。

参照元: [{'source': 'docs/就業規則.txt', 'chunk': 3}]
```

### ノーコード代替: ChatGPTのMyGPTs / Dify

コードを書かずにRAGを実現する方法もあります。

| ツール | 方法 | メリット | デメリット |
|--------|------|---------|-----------|
| **ChatGPT MyGPTs** | Knowledge にPDFアップロード | 最も簡単（5分で構築） | カスタマイズ限定的 |
| **Dify** | GUIでRAGパイプラインを構築 | ノーコード、柔軟 | セルフホストが必要 |
| **Coze** | BotにKnowledgeを追加 | 簡単、多プラットフォーム | データの所在に注意 |

### 演習2: MyGPTsで簡易ナレッジボットを作ろう（10分）

**手順（ChatGPT Plus必要）:**

1. ChatGPT → 「GPTを作成」
2. 名前: 「○○社 社内AIアシスタント」
3. Instructions に以下を貼り付け:

```
あなたは○○株式会社の社内AIアシスタントです。
アップロードされた社内ドキュメントの内容に基づいて回答してください。

ルール:
1. ドキュメントに記載のない情報は「社内資料には記載がありません」と回答
```

>
> - MyGPTsにアップロードしたデータはOpenAIのサーバーに保存される

> - アップロード前に「この情報が外部に出ても問題ないか」を必ずチェック

## PART 4: RAGの精度向上テクニック＆運用設計（10分）

### よくある問題と対策

| 問題 | 原因 | 対策 |
|------|------|------|
| 関係ない文書を参照する | チャンクが大きすぎる / 類似度閾値が低い | チャンクサイズ縮小 + 閾値引き上げ |
| 正しい情報があるのにヒットしない | チャンク分割で情報が断裂 | オーバーラップ増加 + 親子チャンク |
| 回答が一般論になる | 検索結果のランキングが悪い | リランキング（Reranker）の導入 |
| 古い情報で回答する | ドキュメントが更新されていない | 定期的な再取り込みパイプライン |
| ハルシネーション（嘘の回答） | LLMが検索結果を無視 | プロンプトで「出典のみから回答」を厳守 |

### 精度向上の5つのテクニック

```
Level 1: チャンク分割の最適化
  → サイズ調整、オーバーラップ、意味単位分割

Level 2: メタデータフィルタリング
  → 「経費に関する質問」→ 経費関連文書のみを検索対象に

Level 3: ハイブリッド検索
  → ベクトル検索 + キーワード検索の組み合わせ

Level 4: リランキング
  → 検索結果をCohereのRerankerで並べ替え

Level 5: 質問の書き換え（Query Rewriting）
  → 「有休って取れる？」→「有給休暇の取得条件と申請方法」に変換
```

### RAGの回答品質を評価するプロンプト

```
以下のRAGシステムの回答を評価してください。

【質問】有給休暇の申請方法は？
【検索されたドキュメント】（チャンク3件）
【RAGの回答】（生成された回答）

評価基準:
1. 正確性（ドキュメントの内容と一致しているか）: /5
2. 網羅性（必要な情報が含まれているか）: /5
3. 簡潔性（余計な情報がないか）: /5
4. 出典明示（どのドキュメントを参照したか明記されているか）: /5
5. ハルシネーション（ドキュメントにない情報を生成していないか）: /5

合計: /25
改善提案:
```
