### スライド3: なぜロードマップが必要なのか（1:30〜3:00）
**レイアウト**: 上部に問いかけ、下部に「失敗パターン3つ」を横並びカード

> ロードマップなしで進めると、3つの典型的な失敗パターンに陥ります。
>
> 1つ目は「ツール乱立」。部署ごとにバラバラなAIツールを入れて、ノウハウが共有されない。2つ目は「PoC止まり」。実験はしたけど本番業務に乗らない。3つ目は「トップダウン空振り」。社長が「AI使え」と言ったけど現場が動かない。
>
> これ全部、計画がないことが原因です。

- 失敗パターン1: ツール乱立 — 部署ごとにバラバラ導入、ノウハウが散逸
- 失敗パターン2: PoC止まり — 実験で終わり、本番業務に移行できない
- 失敗パターン3: トップダウン空振り — 号令だけで現場のフォローなし
- **共通原因**: 段階的な計画とマイルストーンがない

### スライド4: AI導入の3段階フレームワーク（3:00〜5:00）
**レイアウト**: 横向き矢印フロー図（3段階）、各段階の下にキーワード

> AI導入は、大きく3つのフェーズで考えます。
>
> まず「PoCフェーズ」。ここは実験です。小さく試して、効果があるかどうかを検証する。次に「展開フェーズ」。PoCで効果が確認できたものを、他の部署や業務に広げていく。最後に「定着フェーズ」。AIを使うのが当たり前になって、運用ルールも整備されている状態です。
>
> 重要なのは、このフェーズを飛ばさないことです。いきなり全社展開しようとすると、ほぼ確実に失敗します。

### スライド5: Phase 1 — PoCフェーズ（5:00〜8:00）
**レイアウト**: 左に期間・目的、右にチェックリスト形式のタスク一覧

> PoCフェーズでやるべきことを具体的に見ていきましょう。
>
> まず期間は1〜3ヶ月。ここで大事なのは「小さく、速く」です。壮大なプロジェクトを組む必要はありません。
>
> まず、チャンピオン——つまりAI推進の旗振り役を1〜2名決めます。次に、対象業務を1〜2個選びます。選ぶ基準は「効果が見えやすい」「失敗してもダメージが小さい」「データがある程度整っている」の3つです。
>
> 具体的には、議事録の自動要約、問い合わせメールの下書き生成、日報・週報のまとめ、こういった定型的だけど時間がかかる業務がベストです。
>
> そしてPoCの終わりに、必ず効果測定をします。削減時間、品質、ユーザーの満足度。数字で語れるようにしておくのが次のフェーズへの切符になります。

**期間**: 1〜3ヶ月
**目的**: 小さく試して、効果を数字で確認する

| ステップ | 内容 | 期間目安 |
|---------|------|---------|
| 1. チャンピオン選定 | AI推進の旗振り役を1〜2名決める | 1週目 |
| 2. 対象業務の選定 | 効果が見えやすく、リスクが低い業務を1〜2個 | 1〜2週目 |
| 3. ツール選定・設定 | 業務に合ったAIツールを選び、環境構築 | 2〜3週目 |
| 4. 試行運用 | チャンピオンが実業務で使用 | 3〜8週目 |
| 5. 効果測定 | 削減時間・品質・満足度を数値化 | 8〜12週目 |

**業務選定の3基準**:
- 効果が見えやすい（時間削減が定量化できる）
- 失敗してもダメージが小さい（社外への影響が限定的）
- データがある程度整っている（すぐ着手できる）

**PoC向き業務の例**: 議事録要約、メール下書き、日報まとめ、FAQ対応、データ集計

### スライド6: Phase 2 — 展開フェーズ（8:00〜11:00）
**レイアウト**: 上部にフェーズ目的、中央に展開ステップの横フロー

> PoCで「これは使える」と確認できたら、次は展開フェーズです。期間は3〜6ヶ月。
>
> ここでのポイントは3つあります。
>
> 1つ目、横展開の優先順位をつけること。PoCで成功した業務と「構造が似ている」業務から広げます。いきなり全然違う業務に適用しようとすると、また1からPoCをやり直すことになります。
>
> 2つ目、ルールと教育を整備すること。誰が使っていい、どこまでAIに任せていい、出力は必ず人がチェックする、こういった運用ルールを明文化します。そして利用者向けの研修を行います。
>
> 3つ目、社内での成功事例を発信すること。「営業部でこれだけ時間が削減できた」という具体的な数字を社内に共有する。これが他部署の「うちもやりたい」を生み出します。

**期間**: 3〜6ヶ月
**目的**: 成功パターンを他部署・他業務に広げる

| ステップ | 内容 |
|---------|------|
| 1. 横展開の優先順位付け | PoCと構造が似ている業務から展開 |
| 2. 利用ガイドラインの策定 | 利用範囲、チェック体制、禁止事項を明文化 |
| 3. 社内研修の実施 | 部署別の利用方法トレーニング |
| 4. 成功事例の社内発信 | 数字付きの成果を全社共有 |
| 5. サポート体制の構築 | 問い合わせ窓口、FAQ、チャットチャネル |

### スライド7: Phase 3 — 定着フェーズ（11:00〜13:30）
**レイアウト**: 中央に「自走する組織」のイメージ図（サイクル図）

> 最後の定着フェーズ。期間は6ヶ月〜1年。
>
> ここでのゴールは「特定の誰かが旗を振らなくても、AIが当たり前に使われている状態」を作ることです。
>
> 具体的には、業務プロセスそのものにAIが組み込まれている。新しい人が入ってきたら、オンボーディングの中にAIツールの使い方が含まれている。そして定期的に利用状況をモニタリングして、改善サイクルが回っている。
>
> この段階まで来ると、現場から「この業務もAIでできないか？」という声が自然に上がるようになります。これが本当の意味での「AI導入の成功」です。

**期間**: 6ヶ月〜1年
**目的**: AIの利用が業務プロセスに組み込まれ、自走する状態

- 業務プロセスへの標準組み込み
- オンボーディングにAI研修を含める
- 利用状況の定期モニタリング（月次）
- 改善サイクルの運用（PDCA）
- 新規活用テーマの自発的な提案が現場から上がる

**定着の指標**:
- 全社のAIツール月間利用率 > 70%
- 現場発の活用提案が月1件以上
- AI関連の問い合わせが減少傾向（自己解決力向上）

### スライド9: リソース計画（16:00〜18:30）
**レイアウト**: 表形式（フェーズ別の人員・予算・ツール）

> 計画を立てたら、次はリソースです。「誰が」「いくらで」「何を使って」やるのかを明確にしましょう。
>
> PoCフェーズでは、チャンピオン1〜2名の工数が週のうち20%程度。ツール費用は月額1〜5万円程度。この段階では大きな投資は不要です。
>
> 展開フェーズになると、推進チームとして2〜3名。ツール費用も利用者数に応じてスケールします。教育費用も発生します。
>
> 定着フェーズでは、専任の推進担当は減らせますが、モニタリングと改善の工数は継続的に必要です。
>
> この表をそのままロードマップに組み込んでください。経営層は「いくらかかるのか」を最初に聞きますから。

| リソース | PoCフェーズ | 展開フェーズ | 定着フェーズ |
|---------|-----------|-----------|-----------|
| 人員 | チャンピオン1〜2名（兼務・工数20%） | 推進チーム2〜3名 + 部署担当者 | 推進担当1名 + 各部署オーナー |
| ツール費 | 月1〜5万円 | 月5〜30万円 | 月10〜50万円（利用者数に比例） |
| 教育費 | 外部研修1回 | 部署別研修3〜5回 | 年次更新研修 |
| 外部支援 | 必要に応じてコンサル | ガイドライン策定支援 | — |

### スライド10: リスク管理（18:30〜20:30）
**レイアウト**: リスクマトリクス（影響度×発生確率）の2x2マトリクス + 対策表

> どんな計画にもリスクはあります。AI導入で特に注意すべきリスクを4つ挙げます。
>
> 1つ目、セキュリティリスク。社内の機密情報をAIに入力してしまうリスク。対策は利用ガイドラインと入力禁止事項の明確化。
>
> 2つ目、現場の抵抗。「自分の仕事がなくなるんじゃないか」という不安。対策は丁寧なコミュニケーションと、AIは仕事を奪うのではなく「つまらない作業から解放する」というメッセージング。
>
> 3つ目、ハルシネーション。AIが間違った情報を生成するリスク。対策は必ず人間がチェックするルールの徹底。
>
> 4つ目、コスト超過。想定以上に利用が広がって費用が膨らむ。対策は利用状況のモニタリングと予算上限の設定。

| リスク | 影響度 | 対策 |
|-------|-------|------|
| セキュリティ（情報漏洩） | 高 | 利用ガイドライン策定、入力禁止事項の明確化、DLP設定 |
| 現場の抵抗 | 中 | 丁寧なコミュニケーション、成功体験の共有、「解放」の文脈 |
| ハルシネーション（誤情報） | 中〜高 | 人間チェック必須ルール、用途別のリスク分類 |
| コスト超過 | 中 | 月次モニタリング、予算上限設定、段階的スケール |
| ベンダーロックイン | 低〜中 | マルチツール戦略、API標準活用 |

### スライド11: 経営会議での説明の仕方（20:30〜23:00）
**レイアウト**: 「経営報告スライドのサンプル構成」を枠線で示す

> さて、ここまでの内容を経営会議でどう説明するかです。これ、すごく大事です。
>
> 経営層が知りたいのは3つだけ。「何のためにやるのか」「いくらかかるのか」「いつ効果が出るのか」。これに答える構成で説明しましょう。
>
> まず1枚目に「課題と目的」。うちの会社のこの業務に、これだけの非効率がある。AIで解決できる可能性がある。
>
> 2枚目に「ロードマップ」。今日作ったフレームワークをそのまま使います。3段階のフェーズと、各マイルストーン。
>
> 3枚目に「投資と効果」。Phase 1のコストはこれだけ。期待される効果はこれだけ。ROIの詳細は次回（Level 3-12で扱います）。
>
> 4枚目に「リスクと対策」。想定されるリスクと、その対処法。
>
> 5枚目に「次のアクション」。承認いただければ、来月からこれをやります。
>
> この5枚で十分です。20枚も30枚も作る必要はありません。

**経営報告スライド構成（5枚）**:

1. **課題と目的** — 現状の非効率 → AI活用の機会
2. **ロードマップ** — 3段階フレームワーク + マイルストーン
3. **投資と効果** — フェーズ別コスト + 期待効果（概算）
4. **リスクと対策** — 主要リスク4つと対処法
5. **Next Action** — 承認後のアクションプラン（具体的な日程付き）

**Tips**:
- 1枚あたりの情報量は「30秒で読める量」に抑える
- 数字は必ず入れる（「効率化」ではなく「月20時間削減」）
- 質問されそうなポイントは補足スライド（Appendix）に用意
