### スライド2: AIの弱点（0:30〜3:00）
**レイアウト**: 左に「AIが知っていること」、右に「AIが知らないこと」の対比

> 今日は、AIの最大の弱点とその解決策についてお話しします。
>
> ChatGPTもClaudeも、インターネット上の膨大な知識で訓練されています。だから一般的な質問には的確に答えられる。
>
> でも、こんな質問には答えられません。「うちの会社の就業規則で、有給休暇は何日？」「先月の営業会議で決まったことは？」「この製品の最新マニュアルの3ページ目に何が書いてある？」
>
> 当たり前ですよね。AIはあなたの会社の社内文書を読んだことがない。
>
> じゃあどうするか。「AIに社内文書を読ませる」んです。この技術を**RAG**（ラグ）と呼びます。Retrieval-Augmented Generation——検索で補強した生成。

**AIが知っていること vs 知らないこと**:

| AIが知っていること | AIが知らないこと |
|-----------------|----------------|
| 一般的なビジネス知識 | あなたの会社の就業規則 |
| プログラミング言語の文法 | 先月の議事録の内容 |
| 世界の歴史・科学 | 社内製品のマニュアル |
| 法律の一般論 | あなたの会社の契約書の詳細 |
| 公開されている企業情報 | 社内のノウハウ・暗黙知 |

### スライド3: RAGとは（3:00〜6:00）
**レイアウト**: RAGの仕組みを3ステップのフロー図で図解

> RAGの仕組みを3ステップで説明します。
>
> ステップ1、「蓄積」。社内文書をAIが検索できる形に変換して保存する。就業規則、マニュアル、議事録、FAQ、何でもいい。
>
> ステップ2、「検索」。ユーザーが質問すると、まずその質問に関連する文書を検索して引っ張ってくる。これは通常のGoogle検索と似ていますが、「意味の近さ」で検索するのがポイント。キーワード一致ではなく、意味が似ている文書を見つけ出す。
>
> ステップ3、「生成」。検索で見つかった文書をAIに渡して、「この情報をもとに質問に答えて」と指示する。AIは渡された文書の内容を踏まえて回答を生成する。
>
> つまり、AIが「全部知っている」のではなく、「質問のたびに関連文書を検索して読んでから答える」。これがRAGの本質です。図書館の司書みたいなものです。

```
┌──────────┐    ┌──────────┐    ┌──────────┐
│ ①蓄積    │    │ ②検索    │    │ ③生成    │
│          │    │          │    │          │
│ 社内文書を │───▶│ 質問に関連 │───▶│ 検索結果を │
│ 検索可能な │    │ する文書を │    │ AIに渡して │
│ 形に変換  │    │ 引っ張る  │    │ 回答を生成 │
└──────────┘    └──────────┘    └──────────┘
     ↑                               │
     │          ┌──────────┐         │
     └──────────│ ユーザーの │◀────────┘
                │   質問    │
                └──────────┘
```

**RAGのメリット**:
- AIを再学習させる必要がない（コスト大幅削減）
- 文書を追加するだけでAIの知識が増える
- 回答の根拠（出典）を示せる
- 最新情報に常に対応できる

### スライド4: なぜファインチューニングではなくRAGか（6:00〜8:00）
**レイアウト**: RAG vs ファインチューニングの比較表

> ここで「ファインチューニング」という方法との違いを説明しておきます。
>
> ファインチューニングは、AIモデルそのものを社内データで再学習させる方法。これは確かに効果的ですが、コストが高く、データが変わるたびに再学習が必要。技術的なハードルも高い。
>
> RAGは、モデルはそのまま。質問のたびに文書を検索して渡すだけ。文書が変わったら、保存データを更新するだけ。圧倒的に手軽でコストも低い。
>
> 多くの企業では、まずRAGから始めて、本当に必要な場合にだけファインチューニングを検討する、という順序が正解です。

| 比較項目 | RAG | ファインチューニング |
|---------|-----|-----------------|
| コスト | 低（文書の保存＋検索費用） | 高（GPU利用料・学習時間） |
| 更新の容易さ | 文書を追加/差し替えるだけ | 再学習が必要 |
| 技術ハードル | 中〜低 | 高 |
| 回答の根拠 | 出典を示せる | 示しにくい |
| 向いている用途 | 社内FAQ、マニュアル参照 | 特定ドメインの専門回答 |
| おすすめ | まずはこちらから | 必要に応じて後から |

### スライド5: 実装パターン1 — 簡易版: ファイルアップロード（8:00〜12:00）
**レイアウト**: ツールの画面キャプチャ + 手順

> 3つの実装パターンを紹介します。簡単な順から。
>
> パターン1は「ファイルアップロード」。最も簡単です。
>
> ChatGPT（有料版）やClaude（有料版）には、ファイルをアップロードして「この文書を読んで質問に答えて」と指示できる機能があります。これも広い意味ではRAGの一種。
>
> 例えば、就業規則のPDFをClaudeにアップロードして「有給休暇の取得ルールを教えて」と聞く。Claudeは文書を読んで、正確に回答してくれます。
>
> メリットは設定がゼロなこと。デメリットは毎回ファイルをアップロードする手間、ファイルサイズに制限がある、複数人で共有できない。
>
> 個人利用や小規模なチームなら、これで十分な場合も多いです。

**パターン1: ファイルアップロード型**

| 項目 | 内容 |
|------|------|
| 難易度 | 低（設定不要） |
| 対応ツール | Claude Pro, ChatGPT Plus, Gemini Advanced |
| 対応ファイル | PDF, Word, Excel, CSV, テキスト |
| ファイル上限 | ツールにより異なる（Claude: 約10ファイル） |
| 向いている用途 | 個人利用、少数ファイルの参照 |

**実演**:
```
1. Claude.aiにログイン
2. 「就業規則.pdf」をドラッグ＆ドロップ
3. プロンプト:
   「アップロードした就業規則を読んで、以下の質問に答えてください。
    回答には該当する条項番号も記載してください。
    質問: 有給休暇は入社何ヶ月から取得できますか？」
```

**メリット**: 設定ゼロ、すぐ使える
**デメリット**: 毎回アップロード必要、サイズ制限あり、共有不可

### スライド6: 実装パターン2 — 中級: ベクトルDB（12:00〜17:00）
**レイアウト**: ベクトル検索の仕組み図 + ツール紹介

> パターン2は「ベクトルDB」を使う方法。ここから少し技術的になります。
>
> ベクトルDBとは何か。テキストを「数値のベクトル（方向を持つ数値の列）」に変換して保存するデータベースです。難しく聞こえますが、やっていることは「文章の意味を数値化して、意味が近いもの同士を検索できるようにする」だけです。
>
> 例えば、「有給休暇の取得方法」という質問と、就業規則の「年次有給休暇に関する規定」は、キーワードが完全に一致しなくても「意味が近い」と判定できる。これがベクトル検索の強みです。
>
> 最近は、これを簡単にできるサービスが出ています。Pinecone、Weaviate、Chroma。あるいは、NotionAI、DocsBot、CustomGPTのようなサービスは、文書をアップロードするだけでベクトル化と検索を自動でやってくれます。

**ベクトル検索のイメージ**:
```
「有給休暇の取得方法」→ [0.23, 0.87, -0.12, ...]（ベクトル化）
                                ↓ 意味の近さで検索
就業規則の各条文 → それぞれベクトル化されて保存済み
                                ↓ 最も近い文書を取得
「第15条 年次有給休暇」がヒット
```

**主要なベクトルDB / RAGサービス**:

| ツール | タイプ | 特徴 | 月額目安 |
|-------|-------|------|---------|
| Pinecone | クラウドDB | 最もメジャー、高速 | 無料〜$70 |
| Weaviate | クラウド/OSS | オープンソースも選べる | 無料〜 |
| Chroma | OSS（ローカル） | ローカル実行可、Python | 無料 |
| DocsBot | SaaS | ノーコードでRAG構築 | $19〜 |
| CustomGPT | SaaS | GPT + 社内文書 | $49〜 |
| Dify | OSS/SaaS | RAG + ワークフロー | 無料〜 |

**中級パターンの構築フロー**:
1. 社内文書（PDF, Word等）を準備
2. RAGサービスにアップロード（自動でベクトル化）
3. チャットインターフェースを設定
4. ユーザーが質問→自動で関連文書を検索→AIが回答生成

### スライド7: 実装パターン3 — 上級: 本格RAGシステム（17:00〜20:00）
**レイアウト**: システム構成図（LangChain/LlamaIndex + ベクトルDB + AI API）

> パターン3は「本格的なRAGシステム」の構築。エンジニアが関与するレベルです。
>
> LangChainやLlamaIndexといったフレームワークを使って、文書の取り込み、チャンク分割、ベクトル化、検索、回答生成の全工程をカスタマイズします。
>
> なぜここまでやるのか。パターン2で対応できない要件がある場合です。例えば、検索の精度を極限まで高めたい。社内システムとリアルタイム連携したい。セキュリティ要件が厳しくデータを外部に出せない。大量の文書（数万件以上）を扱う必要がある。
>
> 正直、多くの企業ではパターン1かパターン2で十分です。パターン3が必要になるのは、AI活用が組織に定着してから。

**本格RAGのシステム構成**:
```
┌─────────────────────────────────────────┐
│            社内ネットワーク              │
│                                         │
│  ┌──────────┐  ┌──────────┐  ┌────────┐ │
│  │ 文書管理  │  │ チャンク  │  │ベクトル │ │
│  │ システム  │→│  分割    │→│  DB    │ │
│  │(SharePoint│  │ (512文字  │  │(Pinecone│ │
│  │ Box等)   │  │  単位)   │  │ Chroma) │ │
│  └──────────┘  └──────────┘  └───┬────┘ │
│                                   │      │
│  ┌──────────┐  ┌──────────┐      │      │
│  │ チャット  │→│ 検索+    │◀─────┘      │
│  │ UI      │  │ AI生成   │              │
│  │(Web/Slack)│  │(LangChain│              │
│  └──────────┘  └──────────┘              │
└─────────────────────────────────────────┘
```

**パターン3が必要になるケース**:
- 文書数が数万件以上
- 検索精度に高い要求がある
- 社内システムとのリアルタイム連携
- データを一切外部に出せないセキュリティ要件
- チャンク分割や検索ロジックをカスタマイズしたい

### スライド8: どのパターンを選ぶか（20:00〜22:00）
**レイアウト**: 判定フローチャート

> 3つのパターンのどれを選ぶか。判断基準はシンプルです。

**パターン選定フローチャート**:
```
Q1: 利用者は1〜3人程度？
  YES → パターン1（ファイルアップロード）
  NO ↓

Q2: 文書数は1000件未満？
  YES → パターン2（ベクトルDB / SaaS）
  NO ↓

Q3: エンジニアチームがいる？
  YES → パターン3（本格RAG）
  NO → パターン2 + 外部支援で対応
```

**パターン比較まとめ**:

| 項目 | パターン1 | パターン2 | パターン3 |
|------|---------|---------|---------|
| 難易度 | 低 | 中 | 高 |
| 初期コスト | 0円 | 数千〜数万円 | 数十万〜数百万円 |
| 月額コスト | AI有料プラン代のみ | 数千〜数万円 | 数万〜数十万円 |
| 対応文書数 | 〜10ファイル | 〜数千件 | 数万件以上 |
| 利用者数 | 1〜3人 | チーム〜部署 | 全社 |
| セキュリティ | ツール依存 | SaaS依存 | 完全制御可能 |
| 構築期間 | 即日 | 1〜2週間 | 1〜3ヶ月 |

### スライド9: データ準備のコツ（22:00〜24:00）
**レイアウト**: チェックリスト形式

> どのパターンでも共通して大事なのが「データ準備」です。ゴミを入れればゴミが出る（Garbage In, Garbage Out）。
>
> まず、文書を「きれいにする」。スキャンPDFでOCRが荒いもの、レイアウトが崩れているもの、古くて無効な情報を含むものは、事前に整理してください。
>
> 次に、文書の「粒度」を揃える。100ページの規程集を1ファイルで入れるより、章ごとに分割する方が検索精度が上がります。
>
> そして、「メタデータ」をつける。文書のタイトル、カテゴリ、作成日、最終更新日。これがあると、検索結果のフィルタリングや出典の表示ができます。

**データ準備チェックリスト**:

- [ ] **文書のクリーニング**: OCR品質の確認、不要な装飾の削除
- [ ] **最新性の確認**: 古い・無効な文書は除外またはマーク
- [ ] **適切な粒度で分割**: 章単位やセクション単位に分割
- [ ] **メタデータの付与**: タイトル、カテゴリ、作成日、更新日
- [ ] **重複の排除**: 同じ文書の複数バージョンは最新のみ残す
- [ ] **機密区分の確認**: 機密文書をRAGに含めてよいかセキュリティ担当と確認
- [ ] **テスト質問の準備**: 想定される質問リスト（検証用）

**よくある失敗と対策**:

| 失敗 | 原因 | 対策 |
|------|------|------|
| 検索精度が低い | 文書の粒度が大きすぎる | チャンク分割の見直し |
| 古い情報が出る | 文書の更新管理ができていない | 定期的なデータ更新フロー構築 |
| 機密漏洩 | 権限管理なし | アクセス制御付きRAGの構築 |
| 回答が的外れ | プロンプト設計が不適切 | システムプロンプトの改善 |

### スライド10: まとめと次のアクション（24:00〜25:00）
**レイアウト**: Key Takeaways + Action Items

> 今日のまとめです。
>
> RAGは「AIに社内のことを教える」技術。仕組みは蓄積→検索→生成の3ステップ。実装パターンは3つ。まずはパターン1（ファイルアップロード）から始めて、必要に応じてスケールアップ。データ準備が成否を分ける。
>
> 次のアクションとして、まずClaude Pro（またはChatGPT Plus）で、自社の文書1つをアップロードして質問してみてください。それだけで、RAGの価値が実感できます。

**Key Takeaways**:
1. RAG = AIが質問のたびに関連文書を検索して回答する仕組み
2. ファインチューニングより手軽、低コスト、更新が簡単
3. 3つのパターン: ファイルアップロード → ベクトルDB → 本格RAG
4. 多くの企業はパターン1か2で十分
5. データ準備（クリーニング、粒度、メタデータ）が精度を決める
