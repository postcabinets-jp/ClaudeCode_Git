### スライド3: APIとは何か（2:00〜5:00）
**レイアウト**: レストランの注文に例えた図解

> まず「API」という言葉をわかりやすく説明します。
>
> レストランで考えてください。お客さん（あなた）がウェイター（API）に注文する。ウェイターはキッチン（AIサーバー）に注文を伝える。キッチンが料理（回答）を作って、ウェイターがお客さんに持ってくる。
>
> APIとは、この「ウェイター」です。あなたのシステムとAIの間を取り持つ仲介役。
>
> 技術的に言うと、APIは「あるシステムの機能を、外部から呼び出すための窓口」です。ClaudeのAPIを使えば、あなたのスプレッドシートやアプリから、Claudeの能力を呼び出せる。
>
> 大事なのは、APIを使うのにプログラミングが必須ではないということ。Zapier、Make、n8nのようなノーコードツールを使えば、画面上でポチポチ設定するだけでAPI連携ができます。

**APIの仕組み（レストランモデル）**:
```
お客さん        ウェイター(API)      キッチン(AIサーバー)
  │                │                    │
  │──注文─────────▶│──────────────────▶│
  │                │                    │──調理
  │◀──料理─────────│◀─────────────────│
```

**専門用語の翻訳**:
| 技術用語 | わかりやすく言うと |
|---------|-----------------|
| APIキー | レストランの会員カード（あなたが誰か識別する） |
| リクエスト | 注文（AIにお願いする内容） |
| レスポンス | 料理（AIからの返答） |
| エンドポイント | レストランの住所（AIの窓口のURL） |
| レート制限 | 1分間に注文できる回数の上限 |

### スライド4: Claude API と OpenAI APIの基本（5:00〜9:00）
**レイアウト**: 2列比較（Claude API / OpenAI API）

> 代表的なAI APIを2つ紹介します。Claude APIとOpenAI API（GPT）です。
>
> どちらも基本的な使い方は同じ。「テキストを送ると、AIが処理して結果を返す」。違いは料金体系や得意分野。
>
> Claude APIは、長い文書の処理や正確な指示遵守に強い。OpenAI APIは、画像生成やFunction Callingが充実。用途によって使い分けます。
>
> 料金は「トークン」という単位で課金されます。トークンは大まかに「文字数」と思ってください。日本語1文字が約1〜2トークン。1回のAPI呼び出しで、入力トークン＋出力トークンの合計で課金されます。
>
> 目安として、A4 1ページの文書を要約するリクエストで、だいたい1〜5円程度。月に1000回呼び出しても数千円です。

**Claude API vs OpenAI API 比較**:

| 項目 | Claude API (Anthropic) | OpenAI API (GPT) |
|------|----------------------|-------------------|
| 得意分野 | 長文処理、指示遵守、安全性 | 画像生成、ツール連携、エコシステム |
| 主要モデル | Claude Sonnet / Opus | GPT-4o / GPT-4o-mini |
| 料金体系 | 入力/出力トークン別課金 | 入力/出力トークン別課金 |
| コンソール | console.anthropic.com | platform.openai.com |
| 料金目安 | A4 1ページ要約: 約1〜3円 | A4 1ページ要約: 約1〜5円 |

**トークンの目安**:
- 日本語1文字 ≒ 1〜2トークン
- A4 1ページ ≒ 1,000〜2,000トークン
- 月1,000回の呼び出し ≒ 数千円〜1万円程度

### スライド6: ワークフロー自動化の基本構造（12:00〜14:00）
**レイアウト**: トリガー→処理→アクションのフロー図

> ワークフロー自動化の基本構造は3つの要素です。
>
> 1つ目は「トリガー」。何がきっかけで処理を始めるか。スプレッドシートに新しい行が追加された、メールが届いた、フォームが送信された。
>
> 2つ目は「処理」。トリガーで受け取ったデータをどう加工するか。ここにAI APIを使います。テキストを要約する、分類する、回答を生成する。
>
> 3つ目は「アクション」。処理結果をどこに出力するか。Slackに通知する、スプレッドシートに書き込む、メールを送信する。
>
> この3つを組み合わせるだけ。レゴブロックのように繋げていくイメージです。

```
┌──────────┐    ┌──────────┐    ┌──────────┐
│ トリガー  │───▶│  処理    │───▶│ アクション │
│          │    │ (AI API)  │    │          │
│・新規行追加│    │・テキスト要約│    │・Slack通知 │
│・メール受信│    │・分類     │    │・シート記入 │
│・フォーム  │    │・回答生成  │    │・メール送信 │
└──────────┘    └──────────┘    └──────────┘
```

### スライド7: 実演 — Googleスプレッドシート→AI→Slack通知（14:00〜23:00）
**レイアウト**: 実演のフロー図 + ステップバイステップの手順

> では実演です。こういうシナリオ。
>
> 顧客からの問い合わせがGoogleフォーム経由でスプレッドシートに記録される。新しい問い合わせが入るたびに、AIが内容を分析して緊急度を判定し、担当部署を振り分けて、結果をSlackチャネルに通知する。
>
> これを人手でやると、誰かがスプレッドシートを定期的にチェックして、内容を読んで、判断して、Slackに転記する。1件5分として、1日20件なら100分。
>
> 自動化すれば、問い合わせが入った瞬間に処理が走ります。ゼロ秒。しかも24時間365日。

**実演シナリオ**: 顧客問い合わせの自動分類＋通知

**全体フロー**:
```
Googleフォーム → Googleスプレッドシート → Zapier → Claude API → Slack
    (入力)          (トリガー)         (処理)    (AI分析)   (通知)
```

**ステップ1: Googleスプレッドシートの準備**

| 列 | 内容 |
|----|------|
| A | タイムスタンプ |
| B | 会社名 |
| C | 担当者名 |
| D | 問い合わせ内容 |
| E | （自動記入）緊急度 |
| F | （自動記入）担当部署 |
| G | （自動記入）要約 |

**ステップ2: Zapierでワークフロー作成**

```
1. Trigger: Google Sheets → New Spreadsheet Row
   - スプレッドシートを選択
   - トリガーとなるシートを指定

2. Action 1: Webhooks by Zapier → POST
   - URL: https://api.anthropic.com/v1/messages
   - Headers:
     x-api-key: [あなたのAPIキー]
     anthropic-version: 2023-06-01
     content-type: application/json
   - Body:
     {
       "model": "claude-sonnet-4-20250514",
       "max_tokens": 500,
       "messages": [{
         "role": "user",
         "content": "以下の顧客問い合わせを分析してください。\n\n問い合わせ内容: {{D列の値}}\n\n以下のJSON形式で回答してください:\n{\"urgency\": \"高/中/低\", \"department\": \"営業/技術/総務/その他\", \"summary\": \"30字以内の要約\"}"
       }]
     }

3. Action 2: Google Sheets → Update Spreadsheet Row
   - E列にurgency、F列にdepartment、G列にsummaryを書き込み

4. Action 3: Slack → Send Channel Message
   - チャネル: #customer-inquiries
   - メッセージ:
     📩 新規問い合わせ
     会社: {{B列}}
     担当: {{C列}}
     緊急度: {{urgency}}
     部署: {{department}}
     要約: {{summary}}
```

> これで完成です。テストしてみましょう。スプレッドシートに1行追加すると……はい、数秒でSlackに通知が来ました。緊急度は「高」、担当部署は「技術」、要約も的確です。

**ステップ3: テストと確認**
- スプレッドシートにテストデータを1行追加
- Zapierのログで処理の流れを確認
- Slackの通知内容を確認

### スライド9: セキュリティの注意点（25:30〜28:00）
**レイアウト**: 赤枠の警告エリア + チェックリスト

> API連携で一番注意すべきはセキュリティです。
>
> まず、APIキーの管理。Zapierのようなツールの中で設定するAPIキーは、そのツールのセキュリティに依存します。信頼できるツールを使い、不要になったキーは即座に無効化してください。
>
> 次に、データの流れ。どのデータがどこを通るかを把握しておく。例えば、顧客の個人情報がAI APIを経由する場合、AIプロバイダーの利用規約（データの取り扱い）を確認してください。Anthropicの場合、API経由のデータはモデルの学習には使われません。OpenAIも同様のオプトアウトポリシーがあります。
>
> そして、アクセス権限。ワークフローツールに与える権限は最小限にする。Googleスプレッドシートの全権限ではなく、特定のシートだけ、のように。

**セキュリティチェックリスト**:
- [ ] APIキーは環境変数またはツールの安全な保管場所に格納
- [ ] 不要なAPIキーは無効化済み
- [ ] データの流れ（どこを経由するか）を図示している
- [ ] AIプロバイダーのデータ利用規約を確認済み
- [ ] ワークフローツールに与える権限は最小限
- [ ] エラー発生時の通知設定済み
- [ ] 定期的なログレビューを実施

**主要AIプロバイダーのデータポリシー（2025年時点）**:
| プロバイダー | API経由データの学習利用 | データ保持期間 |
|------------|---------------------|-------------|
| Anthropic (Claude) | 使わない（デフォルト） | 30日（安全目的） |
| OpenAI (GPT) | 使わない（APIはオプトアウト済み） | 30日 |
| Google (Gemini) | 無料版は利用あり、有料版はなし | 条件による |
