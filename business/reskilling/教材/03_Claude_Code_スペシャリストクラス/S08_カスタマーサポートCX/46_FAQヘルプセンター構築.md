# 46. FAQ・ヘルプセンター構築
**Claude Code スペシャリストクラス — S08 カスタマーサポート・CX**
**POST CABINETS Inc.**

## この講義で学ぶこと

- 顧客からの問い合わせデータを元に、FAQ記事を自動生成する方法
- カテゴリ分類・階層構造の設計手法
- 検索最適化（SEO・サイト内検索）のポイント
- 関連記事リンク設計による自己解決率の向上
- FAQ記事の定期メンテナンス運用フロー

## 業務の概要

ヘルプセンター（FAQ）は、顧客が自力で問題を解決できる「セルフサービス」の要である。適切に構築されたFAQは、問い合わせ件数を30〜50%削減し、サポートチームの負荷を大幅に軽減する。

主な業務フロー:
1. 問い合わせデータの収集・分析（頻出質問の特定）
2. FAQ記事の起草・レビュー・公開
3. カテゴリ体系の設計と記事の分類
4. 検索キーワードの最適化
5. 関連記事リンクの設計
6. 定期的なメンテナンス（古い記事の更新・削除）

## 従来の課題

| 課題 | 影響 |
|------|------|
| FAQ記事の作成に時間がかかる | 新機能リリースに記事が追いつかない |
| カテゴリ分類が属人的 | 顧客が目的の記事にたどり着けない |
| 検索精度が低い | 記事があるのにヒットしない |
| メンテナンスが後回し | 古い情報が残り、かえって混乱を招く |
| 多言語対応の工数が膨大 | 海外顧客への対応が遅れる |

## Claudeによる自動化・効率化

### 1. 問い合わせデータからFAQ候補を自動抽出
過去のチケットデータを分析し、頻出する質問パターンを自動的に抽出・グルーピングする。

### 2. FAQ記事の自動生成
抽出された質問に対して、社内ドキュメントやマニュアルを参照しながら回答文を自動生成する。

### 3. カテゴリ体系の自動設計
記事内容を分析し、最適なカテゴリ階層を提案する。

### 4. 検索キーワード・タグの自動付与
記事の内容から、顧客が使いそうな検索キーワードを自動的に抽出・付与する。

### 5. 関連記事リンクの自動提案
記事間の関連性を分析し、「あわせて読みたい」リンクを自動設計する。

### 6. 定期メンテナンスレポートの生成
アクセスデータや顧客フィードバックを元に、更新が必要な記事を自動で特定する。

## プロンプト例（コピペで使えるレベル）

### プロンプト1: 問い合わせデータからFAQ候補を抽出

```
あなたはカスタマーサポートのFAQスペシャリストです。
以下の問い合わせデータ（直近3ヶ月分）を分析し、FAQ記事として作成すべき質問トップ20を抽出してください。

【出力フォーマット】
| 優先度 | 質問（顧客目線の表現） | 件数目安 | 推奨カテゴリ |
|--------|----------------------|---------|------------|

【分析の観点】
- 同じ意味だが異なる表現の質問はグルーピングすること
- 「〇〇できない」「〇〇の方法」など、顧客が実際に使う言い回しで統一すること
- 件数が多いものだけでなく、解決に時間がかかるものも優先度を上げること

【問い合わせデータ】
{ここにCSVまたはテキストデータを貼り付け}
```

### プロンプト2: FAQ記事の自動生成

```
以下の質問に対するFAQ記事を作成してください。

【質問】
{顧客の質問を記入}

【参考情報】
{社内マニュアルや仕様書の該当箇所を貼り付け}

【出力フォーマット】
# [質問文をそのままタイトルに]

## 回答
（結論を最初の1〜2文で述べる。その後、手順がある場合はステップ形式で記載）

## 手順（該当する場合）
1. ステップ1
2. ステップ2
3. ステップ3

## 補足情報
- 関連する注意点やTips

## この記事で解決しない場合
お問い合わせフォームへの誘導文

【ルール】
- 専門用語は避け、初心者にもわかる言葉で書く
- 1記事1トピックの原則を守る
- 手順は5ステップ以内に収める
- スクリーンショットの挿入箇所を [画像: 〇〇の画面] で指示する
```

### プロンプト3: カテゴリ体系の設計

```
以下のFAQ記事一覧を分析し、最適なカテゴリ体系を設計してください。

【記事一覧】
{FAQ記事のタイトル一覧を貼り付け}

【出力フォーマット】
## カテゴリ構造（2階層）

### 大カテゴリ1: [名前]
説明: [このカテゴリに含まれる内容の概要]
- 小カテゴリ1-1: [名前]（該当記事数: X件）
- 小カテゴリ1-2: [名前]（該当記事数: X件）

### 大カテゴリ2: [名前]
...

## 設計の根拠
- なぜこの分類にしたのか
- 顧客の利用シナリオに基づく説明

【ルール】
- 大カテゴリは5〜8個に収める
- 小カテゴリは各大カテゴリ内で3〜6個
- 「その他」カテゴリは極力作らない
- 顧客の行動・目的ベースで分類する（機能ベースではなく）
```

### プロンプト4: 検索キーワード・タグの自動付与

```
以下のFAQ記事に対して、検索最適化のためのキーワードとタグを付与してください。

【FAQ記事】
タイトル: {タイトル}
本文: {本文}

【出力フォーマット】
## 検索キーワード（10個）
顧客がこの記事を探す際に入力しそうな検索キーワードを、頻度が高い順に列挙

## タグ（5個以内）
記事の分類に使えるタグ

## メタディスクリプション（120文字以内）
検索結果に表示される要約文

## 関連する言い換え表現
同じ意味だが異なる表現のリスト（表記ゆれ対応）

【ルール】
- 専門用語と一般用語の両方を含める
- 「〇〇できない」「〇〇の方法」「〇〇 エラー」など、トラブル系の表現も含める
- 略称・正式名称の両方を含める
```

### プロンプト5: FAQ記事のメンテナンスレポート

```
以下のFAQ記事リストとアクセスデータを分析し、メンテナンスレポートを作成してください。

【FAQ記事リスト】
{記事タイトル、最終更新日、月間PV、直帰率、サポートチケットへの遷移率を貼り付け}

【出力フォーマット】
## 要更新（緊急）
| 記事 | 理由 | 推奨アクション |
|------|------|---------------|

## 要更新（通常）
| 記事 | 理由 | 推奨アクション |
|------|------|---------------|

## 削除候補
| 記事 | 理由 |
|------|------|

## 新規作成推奨
| トピック | 根拠 |
|---------|------|

## メンテナンスサマリー
- 全体の健全性スコア: X/100
- 前月比の改善点・悪化点

【判定基準】
- 最終更新から6ヶ月以上 → 要レビュー
- 直帰率80%以上 → 内容が不十分の可能性
- チケット遷移率30%以上 → 記事で解決できていない
- PVゼロ → 削除または統合候補
```

### プロンプト6: 関連記事リンクの設計

```
以下のFAQ記事一覧を分析し、各記事に設置すべき「関連記事」リンクを提案してください。

【記事一覧】
{記事ID、タイトル、カテゴリ、本文概要を貼り付け}

【出力フォーマット】
各記事について:
- 記事ID: [ID]
- 関連記事リンク（3〜5件）:
  1. [関連記事ID]: [タイトル] — 関連理由: [なぜ関連するか]
  2. ...

【リンク設計のルール】
- 同じカテゴリ内だけでなく、クロスカテゴリのリンクも含める
- 「次に読むべき記事」の流れを意識する
- 問題→原因→解決のストーリーでつなげる
```

## 注意点

1. **正確性の担保**: Claudeが生成したFAQ記事は必ず担当者がファクトチェックする。特に料金・仕様・法的事項は公式情報と照合すること
2. **顧客の言葉を使う**: 社内用語ではなく、顧客が実際に使う言葉で記事を書く。検索キーワードも同様
3. **個人情報の除外**: 問い合わせデータをClaudeに入力する際は、顧客の個人情報（氏名・メールアドレス等）を必ず削除・マスキングすること
4. **更新タイミング**: 製品アップデート時にはFAQの同時更新を運用フローに組み込む
5. **多言語対応**: 翻訳記事は機械翻訳のまま公開せず、ネイティブチェックを入れる

## まとめ

- FAQ・ヘルプセンターは「作って終わり」ではなく、継続的なメンテナンスが成否を分ける
- Claudeを活用することで、記事の起草・カテゴリ設計・検索最適化・メンテナンスレポートを大幅に効率化できる
- 最終的な品質保証は人間が行い、「正確で見つけやすく、わかりやすい」FAQを維持する
- KPI（自己解決率・チケット削減数）を定期計測し、PDCAを回すことが重要
