**Claude Code スペシャリストクラス — S11 システム開発**

## この講義で学ぶこと

- 基本設計書の構成と各セクションの書き方（アーキテクチャ・画面設計・DB設計・API設計）
- 詳細設計書の構成と記述方法（モジュール設計・処理フロー・テーブル定義）
- 設計レビューチェックリストの活用
- Claudeによる設計ドキュメント作成の効率化

---

## 業務の概要

システム設計は、要件定義で「何を作るか」を決めた後、「どう作るか」を定めるフェーズである。

| フェーズ | 内容 | 主な成果物 |
|----------|------|-----------|
| **基本設計（外部設計）** | システムの全体構造と、ユーザーから見える部分を設計 | アーキテクチャ設計書、画面設計書、DB概念設計、API仕様書 |
| **詳細設計（内部設計）** | プログラマが実装できるレベルまで詳細化 | モジュール設計書、処理フロー、テーブル定義書、IF設計書 |

### 基本設計の主要成果物
1. **アーキテクチャ設計** — システム全体の構成、技術スタック、ネットワーク構成
2. **画面設計** — 画面レイアウト、画面遷移、UI仕様
3. **DB設計** — ER図、テーブル一覧、データ項目定義
4. **API設計** — エンドポイント一覧、リクエスト/レスポンス仕様
5. **バッチ設計** — バッチ処理一覧、スケジュール、依存関係

---

## 従来の課題

1. **設計書のフォーマットが統一されていない** — 書く人によって粒度・形式がバラバラ
2. **テーブル定義の作成が単調で時間がかかる** — 数十〜数百テーブルの定義を手作業
3. **画面遷移の全体像が見えにくい** — 個別の画面設計はあるが、全体の流れが把握困難
4. **API仕様の記述が不十分** — エラーケースやバリデーションルールが漏れる
5. **設計レビューが形式的** — チェック観点が明確でなく、重要な指摘が出ない

---

## Claudeによる自動化・効率化

### 1. アーキテクチャ設計の素案
- 要件（規模・性能・予算）を入力すると、適切なアーキテクチャパターンを提案
- 技術スタックの比較検討を支援

### 2. 画面設計の効率化
- 機能要件から画面一覧・画面遷移図を自動生成
- 画面項目定義のテンプレートを一括作成

### 3. DB設計の自動化
- 業務要件からER図（テキストベース）とテーブル定義書を自動生成
- 正規化の妥当性チェック

### 4. API設計
- 機能一覧からRESTful APIのエンドポイント設計を自動生成
- OpenAPI（Swagger）形式での出力

### 5. 設計レビュー支援
- 設計書をインプットすると、レビュー観点に基づく指摘事項を自動検出

### プロンプト3: DB設計（テーブル定義書）

```
あなたはデータベースエンジニアです。
以下の業務要件からDB設計を行ってください。

【対象業務】[例：受注管理]
【主要エンティティ】
[例：
- 顧客（顧客コード、顧客名、住所、電話番号...）
- 商品（商品コード、商品名、単価、カテゴリ...）
- 受注（受注番号、受注日、顧客、合計金額...）
- 受注明細（受注番号、行番号、商品、数量、金額...）]

以下を作成してください:

1. ER図（Mermaid記法）
   - エンティティ間のリレーション（1:N、N:M等）
   - 主キー・外部キーを明示

2. テーブル一覧
   | テーブル名(論理) | テーブル名(物理) | 概要 | レコード数見込 |

3. テーブル定義書（各テーブル）
   ■ テーブル名: [物理名] / [論理名]
   | No | 論理名 | 物理名 | データ型 | 桁数 | NULL | PK | FK | デフォルト | 備考 |
   | 1 | 受注番号 | order_id | BIGINT | - | NOT NULL | ○ | - | AUTO | |
   | 2 | 顧客ID | customer_id | BIGINT | - | NOT NULL | - | ○ → customers.id | - | |

   インデックス:
   | インデックス名 | カラム | ユニーク | 用途 |

4. 正規化の確認
   - 第3正規形まで満たしているかチェック
   - 意図的な非正規化がある場合、その理由を記述

5. 共通カラム設計
   - 作成日時、更新日時、作成者、更新者、削除フラグ等の方針
```

## 注意点

1. **設計は要件定義の忠実な反映** — 設計フェーズで勝手に要件を変えない。変更が必要なら要件変更管理プロセスを経る
2. **テーブル定義のデータ型は対象DBMSに合わせる** — ClaudeにDBMS（PostgreSQL、MySQL、Oracle等）を明示すること
3. **API設計はセキュリティを最優先に** — 認証・認可・入力検証・レート制限はClaudeに必ず含めるよう指示
4. **設計レビューは複数の目で** — Claudeのレビューは補助的なもの。アーキテクト、DBA、セキュリティ担当の人間によるレビューが必須
5. **過剰設計を避ける** — 規模に見合わない複雑なアーキテクチャは運用コストを増大させる
6. **Mermaid/PlantUMLの記法はツール側で確認** — Claudeが出力した図の記法がツールで正しくレンダリングされるか確認

---

## まとめ

- システム設計書の作成は、Claudeにより初稿作成の工数を大幅に短縮できる
- 特にDB設計（テーブル定義書の大量作成）とAPI設計（エンドポイント・エラーの網羅）で効果が大きい
- アーキテクチャ設計では、要件に基づく複数案の比較検討をClaudeが支援
- 設計レビューチェックリストにより、レビューの質と網羅性を向上
- 最終的な設計判断（技術選定、トレードオフの決定）はアーキテクトやリードエンジニアが行う
