## 業務の概要

テストはシステムが要件通りに動作することを検証する工程である。AI未導入企業のシステム開発では、テストの計画と実行が品質を左右する。

| テストレベル | 内容 | 実施者 |
|-------------|------|--------|
| **単体テスト（UT）** | 個々のプログラム・モジュールの動作確認 | 開発者 |
| **結合テスト（IT）** | モジュール間の連携確認 | 開発チーム |
| **システムテスト（ST）** | システム全体の機能・非機能の検証 | テストチーム |
| **ユーザー受入テスト（UAT）** | 業務観点での最終確認 | 発注者（業務部門） |

### テストタイプ
- **機能テスト** — 機能要件の充足確認
- **性能テスト** — レスポンスタイム、スループット
- **セキュリティテスト** — 脆弱性、権限制御
- **回帰テスト** — 変更による既存機能への影響確認
- **ユーザビリティテスト** — 操作性、使い勝手

---

## 従来の課題

1. **テスト計画が不十分** — いきなりテストケースを書き始め、全体戦略がない
2. **テストケースの網羅性不足** — 正常系は書くが、異常系・境界値が抜ける
3. **テストケース作成に時間がかかる** — 1機能あたり数十〜数百ケース
4. **バグ報告の品質が低い** — 再現手順が不明確で、開発者が再現できない
5. **テスト結果の可視化が弱い** — 進捗や品質状況が経営層に伝わらない

---

## Claudeによる自動化・効率化

### 1. テスト計画書のテンプレート生成
- プロジェクト概要を入力すると、テスト計画書の骨子を自動作成
- テスト範囲・除外範囲の明確化を支援

### 2. テストケースの自動生成
- 機能仕様（画面項目定義、ビジネスルール）からテストケースを網羅的に生成
- 同値分割・境界値分析・デシジョンテーブルの手法を適用

### 3. バグ報告書の構造化
- バグの現象を入力すると、再現手順・期待結果・実際の結果を構造化

### 4. テスト結果の分析
- テスト実行結果データから、品質メトリクスの算出とレポートを生成

```
   - 最大値、最大値+1

4. ビジネスルールテスト
   各ルールの条件分岐について:
   - 条件を満たす場合
   - 条件を満たさない場合

   複数項目の組み合わせで影響がある場合

テストケース数は[50]件以上を目標にしてください。
```

## 注意点

1. **テスト計画はプロジェクト初期に策定** — テスト工程に入ってから計画するのでは遅い。要件定義と並行して検討開始
2. **テストケースは仕様書に基づく** — Claudeに「一般的なテスト」を作らせるのではなく、具体的な仕様書を入力すること
3. **テストデータは本番に近いものを準備** — テストデータの品質がテストの品質を左右する
4. **自動テストと手動テストの使い分け** — 全てを手動テストにすると回帰テストのコストが膨大になる
5. **UATは業務部門が主体** — IT部門がUATを代行してはならない。業務部門の当事者が操作して確認する
6. **テスト結果は客観的に報告** — 「問題ありません」ではなく、データに基づいた品質評価を行う

---

## まとめ

- テスト計画書はプロジェクトの品質戦略を定める重要文書。Claudeでテンプレートを素早く作成
- テストケースの自動生成は、正常系・異常系・境界値を網羅的に洗い出すことで抜け漏れを防止
- バグ報告書の構造化により、再現手順の明確化と修正工数の削減に貢献
- テスト結果報告書は、経営層にも伝わるメトリクスベースの品質評価を実現
- テストの最終判断（リリース可否）は人間が行う。Claudeは「テスト設計の生産性エンジン」として活用
