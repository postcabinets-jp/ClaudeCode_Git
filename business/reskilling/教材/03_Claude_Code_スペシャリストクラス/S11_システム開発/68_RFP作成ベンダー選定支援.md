## 業務の概要

RFP（Request for Proposal）は、システム開発を外部ベンダーに依頼する際に、要件・条件・期待を明文化して提示する文書である。AI未導入企業にとって、初めてのシステム導入プロジェクトでは特に重要な工程となる。

| フェーズ | 内容 | 成果物 |
|----------|------|--------|
| **RFI（情報提供依頼）** | ベンダーの概要情報を収集 | RFI回答一覧 |
| **RFP作成** | 要件・条件をまとめて提示 | RFP文書 |
| **提案受領** | ベンダーからの提案書を受領 | 提案書一覧 |
| **評価・選定** | 提案内容を評価し、ベンダーを選定 | 評価結果報告書 |
| **契約交渉** | 契約条件の調整 | 契約書 |

---

## 従来の課題

1. **RFPの書き方が分からない** — 初めてのシステム導入で、何を書けばよいか不明
2. **要件が曖昧なRFP** — ベンダーが見積もれない、提案内容がバラバラになる
3. **ベンダー評価が主観的** — 「プレゼンが上手だった」で選んでしまう
4. **価格だけで判断** — 安さだけで選び、品質や保守対応で後悔する
5. **契約交渉で不利になる** — IT契約の知識不足で、不利な条件を受け入れてしまう
6. **比較検討の工数が大きい** — 複数ベンダーの提案を公平に比較するのが大変

---

## Claudeによる自動化・効率化

### 1. RFPテンプレートの自動生成
- プロジェクト概要を入力すると、業種・規模に応じたRFPの構成と記述例を生成

### 2. 評価基準の設計
- プロジェクトの重視ポイントに応じた評価項目と配点の提案
- 定性評価を定量化するための基準設計

### 3. 提案書の比較分析
- 複数ベンダーの提案内容を整理・比較するマトリクスを作成
- 提案間の差異ポイントを自動抽出

### 4. 質問リストの自動生成
- 提案内容の不明確な点を検出し、確認質問を自動生成
- プレゼンテーション時の質問リストを作成

### 5. 契約交渉ポイントの整理
- IT契約における一般的な交渉ポイントと注意事項を提供

### プロンプト1: RFPの作成

```
あなたはITコンサルタントです。
以下のプロジェクト情報からRFP（提案依頼書）を作成してください。

【発注企業情報】
- 社名: [例：○○商事株式会社]
- 業種: [例：建設資材の卸売業]
- 従業員数: [例：80名]
- IT部門: [例：専任者なし、総務部の1名が兼任]

【プロジェクト概要】
- 目的: [例：基幹システム（販売管理）の刷新]
- 背景: [例：現行オフコンが保守限界、Excel運用の限界]
- 対象業務: [例：見積→受注→出荷→請求→入金]
- 予算: [例：初期3000万円以内、月額ランニング50万円以内]
- 稼働希望時期: [例：2026年10月]

以下の構成でRFPを作成してください:

1. はじめに
   1.1 RFPの目的
   1.2 提案の手続き（スケジュール、提出先、質問方法）
   1.3 機密保持

2. 会社概要
   2.1 企業概要
   2.2 組織体制
   2.3 現行システム構成

3. プロジェクト概要
   3.1 背景と目的
   3.2 スコープ
   3.3 前提条件・制約条件
   3.4 希望スケジュール

4. 業務要件
   4.1 対象業務と業務フロー概要
   4.2 機能要件一覧
   | 機能 | 概要 | 優先度 |
   4.3 帳票要件
   4.4 外部連携要件

5. 非機能要件
   5.1 性能要件
   5.2 可用性要件
   5.3 セキュリティ要件
   5.4 運用・保守要件

6. 提案にあたっての条件
   6.1 提案費用の範囲
   6.2 成果物一覧
   6.3 プロジェクト体制の要件
   6.4 検収条件

7. 提案書の構成（記載してほしい事項）
   - 会社概要・実績
   - 提案システムの概要
   - 開発手法・プロジェクト管理方針
   - 体制と要員
   - スケジュール
   - 概算見積（内訳付き）
   - 運用・保守計画
   - 類似案件の実績

8. 評価基準（概要）
   | 評価項目 | 配点 |

9. 提案の提出要領
   - 提出期限
   - 提出物
   - プレゼンテーション日程
```

### プロンプト2: ベンダー評価マトリクスの設計

```
あなたはベンダー選定の専門家です。
以下のプロジェクト条件に基づき、ベンダー評価マトリクスを設計してください。

【プロジェクトの特性】
- 種別: [例：基幹システム刷新（パッケージ + カスタマイズ）]
- 予算: [例：3000万円]
- 期間: [例：1年]
- 重視ポイント: [例：業務適合性、保守の安定性、費用の妥当性]
- 自社のIT力: [例：低い（専任者なし）]

以下を作成してください:

1. 評価カテゴリと配点
   | カテゴリ | 配点（計100点） | 評価の視点 |
   | 提案内容 | [点] | 要件の充足度、提案の具体性 |
   | 技術力 | [点] | アーキテクチャ、セキュリティ |
   | 実績 | [点] | 類似業種・規模の実績 |
   | 体制 | [点] | PMの力量、SE体制の安定性 |
   | 費用 | [点] | 初期費用、ランニング費用の妥当性 |
   | 保守・サポート | [点] | 保守体制、SLA、対応力 |
   | 企業の安定性 | [点] | 財務状況、継続性 |
```

## 注意点

1. **RFPは社内の合意を得てから発行** — 経営層の予算承認、業務部門の要件合意を経てからベンダーに提示
2. **公平性を担保する** — 特定ベンダーに有利な条件を書かない。質問・回答は全ベンダーに共有
3. **NDA（秘密保持契約）を先行** — RFPには業務情報が含まれるため、RFP提示前にNDAを締結
4. **価格だけで選ばない** — TCO（5年間の総コスト）で比較し、品質・保守・リスクも含めた総合評価を行う
5. **契約書は専門家に確認** — IT契約の法的リスク（知的財産権、瑕疵担保等）は弁護士の確認を受けること
6. **Claudeの出力は「たたき台」** — 自社の業務要件や社内事情はClaudeが知り得ないため、必ず人間がカスタマイズする

## まとめ

- RFPの作成は、Claudeでテンプレートと構成案を素早く生成し、自社の要件を落とし込む
- ベンダー評価マトリクスにより、主観的な判断を排除し、公平・客観的な選定を実現
- プレゼン質問リストは、提案内容の不明確な点を事前に整理し、効率的な確認を可能に
- 契約交渉ポイントの整理により、IT契約に不慣れな企業でも適切な交渉が可能
- 最終的なベンダー選定と契約判断は経営者が行う。Claudeは「選定プロセスの効率化エンジン」として活用する
