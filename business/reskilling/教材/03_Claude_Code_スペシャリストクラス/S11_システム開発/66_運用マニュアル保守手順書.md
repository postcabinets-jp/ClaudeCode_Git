## 業務の概要

システムは「作って終わり」ではなく、本番稼働後の運用・保守が長期にわたって続く。運用マニュアルと保守手順書は、システムの安定稼働を支える基盤文書である。

| 文書 | 内容 | 利用者 |
|------|------|--------|
| **運用マニュアル** | 日常の運用操作手順（監視、バッチ実行、ユーザー管理等） | 運用担当者 |
| **障害対応手順書** | 障害発生時のエスカレーション、切り分け、復旧手順 | 運用担当者、保守チーム |
| **バックアップ手順書** | データのバックアップとリストアの手順 | 運用担当者 |
| **保守手順書** | パッチ適用、バージョンアップ、設定変更の手順 | 保守チーム |
| **SLA定義書** | サービスレベルの合意事項（稼働率、応答時間等） | 発注者、運用チーム |

---

## 従来の課題

1. **運用マニュアルが作られない** — 開発終了と同時にプロジェクトが解散し、運用文書が未整備
2. **障害対応が属人的** — 「あの人がいないと対応できない」状態
3. **手順書が古い** — システム変更後も手順書が更新されず、実態と乖離
4. **バックアップの復旧テストをしていない** — バックアップは取っているが、リストアを試したことがない
5. **SLAが曖昧** — 「なるべく早く対応」では責任範囲が不明確

## Claudeによる自動化・効率化

### 1. 運用マニュアルの構成設計
- システム構成情報を入力すると、必要な運用手順の一覧と各手順の骨子を自動生成

### 2. 障害対応手順書の作成
- 想定される障害パターンごとの対応フローを自動生成
- エスカレーションマトリクスのテンプレート作成

### 3. バックアップ・リストア手順
- システム構成に応じたバックアップ方式の提案と手順書の作成
- リストア手順とテスト計画の生成

### 4. 定期メンテナンス手順
- システム構成から必要な定期メンテナンス項目を洗い出し、スケジュール化

### 5. SLA定義書の策定
- システム要件からSLA項目と目標値を提案

```
あなたはITサービスマネージャーです。
以下のシステムのSLA（サービスレベル合意書）を策定してください。

【システム概要】
- システム名: [例：販売管理システム]
- サービス提供者: [例：IT部門 / 外部ベンダー]
- サービス利用者: [例：営業部、出荷部門、経理部]
- サービス時間: [例：平日8:00-20:00]

以下の構成で作成してください:

1. SLA概要
   1.1 目的
   1.2 適用範囲
   1.3 契約期間
   1.4 SLAレビュー頻度

2. サービスレベル目標
   | SLA項目 | 指標 | 目標値 | 測定方法 | 報告頻度 |
   | 可用性 | 稼働率 | 99.5%以上 | 監視ツール | 月次 |
   | 性能 | 画面応答時間 | 3秒以内（95%ile） | APM | 月次 |
   | 障害復旧 | MTTR | 4時間以内 | 障害記録 | 都度 |
   | 問い合わせ対応 | 初回応答 | 2時間以内（営業時間内） | チケット | 月次 |
   | セキュリティ | パッチ適用 | 重大: 72時間以内 | パッチ管理台帳 | 月次 |
   | バックアップ | 成功率 | 100% | バックアップログ | 月次 |

3. サービスレベルの計算方法
   - 稼働率 = (サービス時間 - 計画外停止時間) / サービス時間 × 100
   - 計画停止はSLA算定から除外
   - 測定ツールと測定ポイント

4. SLA未達時の対応
   | 未達レベル | 対応 |
   | 1回目 | 原因分析・改善計画の提出 |
   | 2回連続 | サービス改善会議の開催 |
   | 3回連続 | 契約見直しの協議 |

5. 責任分界点
   | 領域 | サービス提供者の責任 | 利用者の責任 |

6. 除外事項（SLA対象外）
   - 計画メンテナンス
   - 不可抗力（災害等）
   - 利用者起因の障害

7. SLA報告書テンプレート
```

## 注意点

1. **運用マニュアルは開発フェーズで作成開始** — 本番稼働直前に慌てて作るのでは不十分。設計・開発と並行して整備する
2. **手順書は実際に手順通りに操作して検証** — Claudeが生成した手順をそのまま使わない。必ず実環境で動作確認する
3. **バックアップは復旧テストまでがセット** — バックアップを取っているだけでは不十分。定期的にリストアテストを実施する
4. **SLAは両者の合意** — 一方的に決めず、サービス提供者と利用者の双方で合意する
5. **手順書の更新を仕組み化** — システム変更時に手順書も更新する運用ルールを決め、チェックリストに含める
6. **セキュリティ情報の取り扱い** — パスワード、IPアドレス等の機密情報は手順書に直接書かず、別管理する

---

## まとめ

- 運用マニュアルと保守手順書は、システムの安定稼働を支える基盤。Claudeで構成設計から詳細手順まで効率的に作成
- 障害対応手順書は、想定障害パターンごとの具体的な対応手順をClaudeが自動生成
- バックアップ・リストア手順は、RPO/RTOに基づいた方式設計から手順書化まで一気通貫で支援
- SLA定義書により、サービスレベルの目標と責任範囲を明確化
- 運用文書は「生きた文書」。システム変更に追従して継続的に更新することが最重要
