## この講義で学ぶこと

- データ移行計画書のテンプレートと策定方法

- マッピング表（旧→新）の作成

---

## 業務の概要

データ移行は、旧システムから新システムへデータを移す作業である。AI未導入企業では、旧システムだけでなく、Excel・紙の帳票からのデータ移行も必要になることが多い。

| フェーズ | 内容 | 成果物 |
|----------|------|--------|
| **計画** | 移行方針・スケジュール・体制の策定 | データ移行計画書 |
| **棚卸し** | 移行対象データの洗い出し | データ棚卸し一覧 |
| **設計** | マッピング・変換ルール・クレンジング基準の設計 | マッピング表、変換仕様書 |
| **開発** | 移行ツール/スクリプトの作成 | 移行プログラム |
| **テスト移行** | 本番データのサブセットで移行リハーサル | テスト移行結果報告書 |
| **本番移行** | 実データの移行実施 | 移行実施報告書 |
| **検証** | 移行後のデータ整合性確認 | 検証結果報告書 |

---

## 従来の課題

1. **移行対象データの把握不足** — 旧システムにどんなデータがどれだけあるか不明
2. **データ品質の問題** — 旧データに重複・欠損・不整合が大量にある
3. **マッピングの複雑さ** — 旧と新でデータ構造が異なり、変換ルールが複雑
4. **テスト移行の不足** — リハーサルなしで本番移行し、大量のデータ不整合が発生
5. **切り戻し計画がない** — 移行失敗時に旧システムに戻す手順が未策定
6. **移行期間の業務影響** — データ凍結期間中の業務運用が未検討

---

## Claudeによる自動化・効率化

### 1. データ移行計画書のテンプレート生成
- プロジェクト情報を入力すると、移行計画書の構成と各セクションの記述案を自動生成

### 2. データ棚卸し支援
- テーブル定義やファイル一覧から移行対象データの一覧を整理
- データ量・重要度による優先順位付けを支援

### 3. マッピング表の作成
- 旧テーブル定義と新テーブル定義を入力すると、マッピング表の素案を自動生成
- 変換ルール（コード変換、結合・分割等）の提案

### 4. データクレンジング基準
- データ品質チェックの項目と基準を自動生成
- クレンジングルール（重複排除、欠損補完等）の提案

### 5. 移行手順書と切り戻し計画
- タイムライン付きの本番移行手順を作成
- 判断ポイントと切り戻し判断基準の設計

### プロンプト4: データクレンジング基準

```
あなたはデータ品質管理の専門家です。
以下のデータに対するクレンジング基準を設計してください。

【対象データ】[例：顧客マスタ（約5万件）]

【データ品質の現状（事前調査結果）】
- 重複: [例：同一顧客が複数のコードで登録されている（推定500件）]
- 欠損: [例：電話番号の未入力が約10%]
- 不整合: [例：郵便番号と住所が不一致のケースあり]
- 形式不統一: [例：電話番号の区切りが半角ハイフン/全角ハイフン/なしが混在]

以下を作成してください:

1. データ品質チェック項目
   | No | チェック項目 | チェック方法（SQL/ルール） | 合格基準 | 対象カラム |
   | 1 | NULL/空白チェック | WHERE column IS NULL OR column = '' | 必須項目は0件 | |
   | 2 | 重複チェック | GROUP BY HAVING COUNT > 1 | 0件 | |
   | 3 | 形式チェック | 正規表現パターンマッチ | 不一致0件 | |
   | 4 | 参照整合性チェック | LEFT JOINで不一致抽出 | 0件 | |
   | 5 | 範囲チェック | WHERE column NOT BETWEEN x AND y | 0件 | |

2. クレンジングルール
   | No | 対象 | ルール | 処理内容 | 優先度 |
   | 1 | 重複レコード | 名寄せ基準 | [基準を記述] | High |
   | 2 | 電話番号 | 形式統一 | ハイフンなしの数字列に統一 | Medium |
   | 3 | 郵便番号 | 形式統一 | 7桁数字（ハイフンなし）に統一 | Medium |
   | 4 | 住所 | 正規化 | 都道府県・市区町村の正式表記に統一 | Low |

3. 名寄せ基準（重複排除）
   | 判定条件 | 重複の確信度 | 対応 |
   | 完全一致（名称+住所+電話） | 確定 | 自動マージ |
   | 部分一致（名称類似+住所一致） | 高 | 目視確認後マージ |
   | 名称のみ一致 | 低 | 目視確認 |

4. クレンジング不可能なデータの扱い
   | ケース | 対応方針 |
   | 修正不可能な欠損 | デフォルト値 or 移行除外 |
   | 判断が必要なもの | 業務部門に確認 |

5. クレンジング実施手順とスケジュール
```

## 注意点

1. **テスト移行は最低3回実施** — 1回目で問題発見、2回目で対策確認、3回目で本番リハーサル
2. **データ凍結期間の業務影響を事前検討** — 凍結中に発生する取引データの取り扱いを決めておく
3. **切り戻し計画は必須** — 移行失敗時に旧システムに戻す手順を必ず準備。「切り戻さない」は計画ではない
4. **業務部門によるデータ検証が不可欠** — 件数合計の一致だけでなく、業務観点での正確性確認が必要
5. **個人情報の取り扱い** — データ移行時の個人情報保護法への対応（テスト環境でのマスキング等）
6. **旧データの保管** — 移行完了後も旧データは一定期間保管する（法令遵守、監査対応）

## まとめ

- データ移行計画書は、Claudeで構成・テンプレートを素早く生成し、プロジェクト固有の情報を落とし込む
- データ棚卸しとクレンジング基準の設計で、移行前にデータ品質を確保
- マッピング表は旧・新のテーブル定義を入力すれば、変換ルールの素案をClaudeが自動生成
- 本番移行手順書はタイムライン形式で、判断ポイントと切り戻し基準を明確化
- データ移行の成否はテスト移行の回数と質で決まる。リハーサルを十分に行うこと
