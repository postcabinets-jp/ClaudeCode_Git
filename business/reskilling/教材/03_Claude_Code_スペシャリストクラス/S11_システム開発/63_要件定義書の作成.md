# 63. 要件定義書の作成
**Claude Code スペシャリストクラス — S11 システム開発**
**POST CABINETS Inc.**

---

## この講義で学ぶこと

- 要件定義書の全体構成と各セクションの書き方
- 機能要件・非機能要件の整理方法
- ユースケース記述と画面遷移図の作成
- ステークホルダーヒアリング質問の設計
- Claudeを活用した要件定義の効率化

---

## 業務の概要

要件定義は「何を作るか」を明確にするフェーズであり、システム開発の成否を左右する最重要工程である。

| 要素 | 内容 |
|------|------|
| **目的** | システムで実現すべきことを関係者間で合意する |
| **成果物** | 要件定義書（RD: Requirements Definition） |
| **参加者** | 発注者（業務部門）、PM、SE、業務コンサルタント |
| **期間** | プロジェクト全体の15〜25%程度 |

### 要件定義書の主要構成

1. プロジェクト概要（背景・目的・スコープ）
2. 業務要件（業務フロー、業務ルール）
3. 機能要件（画面・帳票・バッチ・インターフェース）
4. 非機能要件（性能・可用性・セキュリティ等）
5. 制約条件（技術・予算・スケジュール・組織）
6. 移行要件
7. 運用・保守要件

---

## 従来の課題

1. **要件の曖昧さ** — 「使いやすいシステム」「早く処理できる」等の定性的な表現が残る
2. **要件の漏れ** — 非機能要件や例外処理の検討が不足
3. **ステークホルダー間の認識齟齬** — 同じ言葉を違う意味で使っている
4. **要件の肥大化** — 「あれもこれも」で優先順位が付かない
5. **要件変更の管理** — 口頭での変更が記録されず、スコープクリープが発生

---

## Claudeによる自動化・効率化

### 1. 要件定義書のテンプレート生成
- プロジェクト概要を入力すると、業種・規模に応じた要件定義書の骨子を自動生成

### 2. 機能要件の洗い出し
- 業務フローから必要な画面・機能を自動抽出
- CRUD（登録・参照・更新・削除）マトリクスの生成

### 3. 非機能要件のチェックリスト
- IPAの非機能要求グレード等に基づく網羅的な確認項目を提供

### 4. ユースケース記述
- アクター（利用者）とシステムのやり取りを構造化された形式で記述

### 5. ヒアリング質問設計
- 要件の不明確な箇所を検出し、確認すべき質問を自動生成

## プロンプト例（5つ以上、コピペで使えるレベル）

### プロンプト1: 要件定義書の骨子作成

```
あなたはシステム開発のシニアSEです。
以下のプロジェクト情報から、要件定義書の骨子を作成してください。

【プロジェクト概要】
- プロジェクト名: [例：販売管理システム刷新]
- 背景: [例：現行システム（オフコン）が保守限界を迎え、
  Excel運用が増加。業務効率が低下している]
- 目的: [例：業務効率の向上、データの一元管理、経営判断の迅速化]
- 対象業務: [例：見積→受注→出荷→請求→入金]
- 利用部門: [例：営業部（15名）、出荷部門（8名）、経理部（5名）]
- 予算規模: [例：3000万円]
- 稼働時期: [例：2026年4月]

以下の構成で要件定義書の骨子を作成してください:

1. プロジェクト概要
   1.1 背景と目的
   1.2 スコープ（対象範囲と対象外）
   1.3 前提条件と制約条件
   1.4 プロジェクト体制
   1.5 スケジュール概要

2. 業務要件
   2.1 対象業務一覧
   2.2 業務フロー（To-Be）の概要
   2.3 業務ルール一覧

3. 機能要件
   3.1 機能一覧（サブシステム別）
   3.2 画面一覧
   3.3 帳票一覧
   3.4 バッチ処理一覧
   3.5 外部インターフェース一覧

4. 非機能要件
   4.1 性能要件
   4.2 可用性要件
   4.3 セキュリティ要件
   4.4 拡張性要件
   4.5 運用・保守要件

5. データ要件
   5.1 主要データ一覧
   5.2 データ量見積り
   5.3 データ移行要件

各項目には、具体的な記述例を付けてください。
「要検討」「TBD」の箇所も明示してください。
```

### プロンプト3: 非機能要件の定義

```
あなたはシステムアーキテクトです。
以下のシステムの非機能要件を定義してください。

【システム概要】
- 種別: [例：社内業務システム（Web）]
- 利用者数: [例：約30名]
- 利用時間帯: [例：平日8:00-20:00]
- データ量: [例：年間約10万件の取引データ]

IPAの非機能要求グレードを参考に、以下のカテゴリで要件を定義してください:

1. 可用性
   - 稼働率目標: [例：99.5%以上（月間停止 3.6時間以内）]
   - 計画停止: [保守ウィンドウ]
   - 障害時の復旧目標: RTO=[  ] / RPO=[  ]
   - 冗長化要件

2. 性能・拡張性
   - レスポンスタイム: [画面操作は○秒以内]
   - スループット: [同時接続数、バッチ処理時間]
   - 将来のデータ量・利用者数の増加見込み
   - スケールアップ/スケールアウトの方針

3. 運用・保守性
   - 監視要件
   - バックアップ/リストア要件
   - ログ管理
   - パッチ適用方針

4. 移行性
   - データ移行の範囲と方法
   - 並行運用期間

5. セキュリティ
   - 認証・認可
   - データ暗号化
   - 監査ログ
   - ネットワーク制御

6. システム環境・エコロジー
   - 動作環境（サーバ/クライアント）
   - ネットワーク要件

各項目について、この規模のシステムに適切なレベルを提案してください。
過剰品質にならないよう注意すること。
```

### プロンプト5: ステークホルダーヒアリング質問設計

```
あなたはITコンサルタントです。
以下の要件定義の不明確な点を解消するためのヒアリング質問を設計してください。

【現在の要件定義書（ドラフト）の主要機能】
[機能一覧を貼り付け]

【不明確な点・確認が必要な点】
[例：
- 承認フローの段階数と承認者の条件
- 在庫引当のタイミング（受注時 or 出荷指示時）
- 既存データの移行範囲]

以下のステークホルダーごとにヒアリング質問を設計してください:

1. 経営層（意思決定者）
   - プロジェクトの優先順位に関する質問
   - 予算・スケジュールのコミットメントに関する質問
   - 成功基準の確認
   （質問5つ以上）

2. 業務部門（現場の利用者）
   - 業務ルールの詳細確認
   - 例外処理・イレギュラー対応の確認
   - 使い勝手の要望
   （質問8つ以上）

3. IT部門（運用担当）
   - 既存システムとの連携
   - 運用・保守の体制
   - セキュリティ要件
   （質問5つ以上）

各質問には以下を付記:
- 質問の意図（なぜこの質問が重要か）
- 想定される回答パターンと、それによる設計への影響
```

### 機能要件一覧テンプレート

```
===== 機能要件一覧 =====
文書番号: RD-[コード]-FR    版数: [Ver. X.X]

| 機能ID | サブシステム | 機能名 | 機能概要 | 優先度 | 画面/バッチ/IF | 関連UC | 備考 |
|--------|------------|--------|----------|--------|--------------|--------|------|
| F-001  |            |        |          | Must   | 画面         |        |      |
| F-002  |            |        |          | Must   | 画面         |        |      |
| F-003  |            |        |          | Should | バッチ       |        |      |

優先度定義:
- Must: 必須（これがないとシステムが成立しない）
- Should: 重要（本稼働までに実現すべき）
- Could: あれば望ましい（Phase2以降でもよい）
- Won't: 今回は対象外（将来の検討事項として記録）
```

## 注意点

1. **要件定義は「合意形成」のプロセス** — Claudeで書いた要件定義書をそのまま使うのではなく、ステークホルダーとのレビュー・合意の土台として活用する
2. **機能要件だけでなく非機能要件も** — 性能・セキュリティ・運用の要件は後から追加すると大幅な手戻りになる。最初から検討する
3. **要件の優先順位を必ず付ける** — MoSCoW法（Must/Should/Could/Won't）で分類し、スコープを管理する
4. **数値で定義する** — 「早い」→「3秒以内」、「大量」→「月間10万件」のように定量化する
5. **変更管理の仕組みを最初に決める** — 要件は変わるものという前提で、変更プロセスを合意しておく
6. **Claudeは業務固有のルールを知らない** — 業界の商慣習や社内ルールは必ず人間がインプットすること

---

## まとめ

- 要件定義書の骨子作成は、プロジェクト概要を入力するだけでClaudeが構成案を生成
- 機能要件の詳細化（ユースケース、CRUD、ビジネスルール）まで一気通貫で支援
- 非機能要件はIPAの非機能要求グレードを参考に、システム規模に見合った適切なレベルを提案
- ステークホルダーヒアリングの質問設計により、要件の抜け漏れを早期に防止
- 要件定義書の最終品質は、ステークホルダーとの合意プロセスで決まる。Claudeは「たたき台を高速に作るエンジン」として活用する
