## 業務の概要

データベース設計は、アプリケーションの基盤となるデータ構造を定義する重要な工程である。適切な設計は、データの整合性・検索速度・保守性を大きく左右する。また、日常的なSQL作成業務も、複雑な結合やサブクエリが絡むと時間がかかる作業である。

主な業務フロー:
1. 業務要件からエンティティ（テーブル）を洗い出す
2. エンティティ間のリレーションシップを定義する
3. テーブル定義書を作成する（カラム名・型・制約）
4. 正規化を行い、データの冗長性を排除する
5. インデックスを設計する
6. DDL（テーブル作成SQL）を生成する
7. 業務で必要なクエリ（DML）を作成する

## 従来の課題

| 課題 | 影響 |
|------|------|
| テーブル設計が属人的 | 設計者によって命名規則やデータ型がバラバラ |
| 正規化の判断が難しい | 正規化不足→データ不整合、過正規化→パフォーマンス低下 |
| 複雑なSQLの作成に時間がかかる | 3テーブル以上のJOINやサブクエリで試行錯誤する |
| パフォーマンス問題の原因特定が難しい | スロークエリの改善方法がわからない |
| テーブル定義書の更新が追いつかない | ドキュメントと実態が乖離する |

## Claudeによる自動化・効率化

### 1. 業務要件からテーブル設計を自動提案
業務の説明からエンティティ・リレーションシップを自動的に抽出し、テーブル構造を提案する。

### 2. テーブル定義書とDDLの自動生成
テーブル設計からDDL（CREATE TABLE文）とテーブル定義書を自動生成する。

### 3. SQLクエリの自動生成
自然言語の要件から、適切なSQLクエリを自動生成する。

### 4. 正規化チェックと改善提案
既存のテーブル構造を分析し、正規化の問題点と改善案を提示する。

### 5. パフォーマンス改善提案
スロークエリの分析とインデックス設計の提案を行う。

## 注意点

1. **本番環境での実行注意**: Claudeが生成したSQLは必ずテスト環境で検証してから本番実行すること。特にDDL（ALTER TABLE等）とDELETE/UPDATEは要注意
2. **RDBMS固有の構文**: PostgreSQL・MySQL・SQL Server等で構文が異なる場合がある。対象RDBMSを必ず指定すること
3. **データ量の考慮**: 小規模データでは問題ないSQLが、大規模データでは極端に遅くなることがある。本番のデータ量を前提にテストすること
4. **セキュリティ**: SQLインジェクション対策のため、生成されたSQLをアプリケーションに直接埋め込まず、パラメータバインドを使用すること
5. **バックアップ**: テーブル構造の変更やデータ移行の前には必ずバックアップを取得すること

## まとめ

- データベース設計は「業務要件→エンティティ抽出→正規化→物理設計」の順で進める
- Claudeを活用することで、テーブル設計・DDL生成・SQLクエリ作成・正規化チェック・パフォーマンス分析を効率化できる
- 生成されたSQLは必ず検証してから実行する。特に本番環境では慎重に
- パフォーマンスは設計段階で考慮する。後からの修正はコストが高い
