# 54. システム要件定義・RFP作成
**Claude Code スペシャリストクラス — S09 IT・情報システム**
**POST CABINETS Inc.**

## この講義で学ぶこと

- 要件定義書の構成と各セクションの書き方
- 機能要件・非機能要件の整理手法
- RFP（提案依頼書）の全体構成と作成プロセス
- ベンダー評価基準の設計
- 要件の優先度付けとスコープ管理

## 業務の概要

システム要件定義は、業務課題を解決するシステムに「何が必要か」を定義するプロセスである。RFP（Request for Proposal）は、定義した要件をベンダーに提示し、提案を求める文書である。要件定義の品質がプロジェクトの成否を大きく左右する。

主な業務フロー:
1. 業務課題・目的の明確化
2. 現行業務フローの整理（As-Is）
3. あるべき業務フローの設計（To-Be）
4. 機能要件の洗い出し
5. 非機能要件の定義
6. RFPの作成・配布
7. ベンダー提案の評価・選定

## 従来の課題

| 課題 | 影響 |
|------|------|
| 要件が曖昧・漏れがある | 開発後に「思っていたのと違う」が発生する |
| 非機能要件が軽視される | パフォーマンスやセキュリティの問題が後から発覚 |
| RFPの書き方がわからない | ベンダーから適切な提案を引き出せない |
| 要件の優先度が不明確 | 予算・期間内に収まらず、全体が遅延する |
| ベンダー評価が属人的 | 「なんとなく良さそう」で選定してしまう |

## Claudeによる自動化・効率化

### 1. 業務ヒアリング結果の構造化
散発的なヒアリングメモを要件定義書の構造に整理する。

### 2. 機能要件・非機能要件の網羅的な洗い出し
業種・システム種別に応じた典型的な要件リストを提示し、抜け漏れを防ぐ。

### 3. RFPのドラフト自動生成
要件定義の結果を元に、RFPの全セクションを自動生成する。

### 4. ベンダー評価基準の設計
客観的な評価基準と採点シートを自動生成する。

### 5. 要件の優先度付け支援
MoSCoW法やKano分析を適用し、要件の優先度を整理する。

```markdown
# 要件管理表

| ID | 要件名 | 種別 | カテゴリ | 説明 | 優先度 | ステータス | 担当 | 備考 |
|----|--------|------|---------|------|--------|----------|------|------|
| FR-001 | | 機能 | | | Must | 未着手 | | |
| FR-002 | | 機能 | | | Should | | | |
| NFR-001 | | 非機能 | 性能 | | Must | | | |
| NFR-002 | | 非機能 | セキュリティ | | Must | | | |

## ステータス定義
- 未着手: 要件として登録済み
- 確認中: ステークホルダーと確認中
- 確定: 合意済み
- 変更: 変更が発生
- 取消: 要件から除外
```

## 注意点

1. **要件の「完璧さ」を求めすぎない**: 初期段階で100%の要件を定義するのは不可能。アジャイル的に段階的に詳細化する発想も持つこと
2. **ステークホルダーの合意**: 要件はClaudeが作るものではなく、ステークホルダーが合意するもの。Claudeの出力はあくまで叩き台
3. **非機能要件の具体性**: 「高速であること」ではなく「画面表示3秒以内」のように、測定可能な数値で定義すること
4. **RFPの公平性**: 特定ベンダーに有利な要件を入れないこと。公平な競争環境を確保する
5. **見積もりの信頼性**: Claudeが出す工数・費用の概算は参考値。実際の見積もりはベンダーから取得すること

## まとめ

- 要件定義は「何を作るか」を決める最重要プロセス。ここでの曖昧さが後工程のすべてに影響する
- Claudeを活用することで、ヒアリング結果の構造化・要件の網羅的な洗い出し・RFPドラフト・評価基準の設計を効率化できる
- 機能要件だけでなく非機能要件（性能・セキュリティ・運用）を必ず定義すること
- 要件の優先度付けとフェーズ分割により、予算・期間の制約の中で最大の価値を実現する
