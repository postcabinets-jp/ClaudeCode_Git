## 業務の概要

### 社内通達とは

社内通達とは、会社から従業員に対して公式に情報を伝達する文書です。規程の改定通知、人事異動、制度変更、注意喚起など、全従業員または特定の対象者に向けて発信されます。

### 社内通達の種類

| 種類 | 目的 | 例 |
|------|------|-----|
| **人事通達** | 人事異動・組織変更の周知 | 異動発令、昇格発令、組織改編 |
| **制度通知** | 制度の新設・変更の周知 | 就業規則改定、福利厚生制度の変更 |
| **業務通知** | 業務上の指示・依頼 | 棚卸し実施要領、年末調整の案内 |
| **注意喚起** | リスク・コンプライアンスの啓発 | 情報セキュリティ注意喚起、ハラスメント防止 |
| **イベント案内** | 社内行事の案内 | 全社会議、忘年会、健康診断 |

### 業務マニュアル・SOPとは

**業務マニュアル**：特定の業務全体の進め方を体系的にまとめた文書。
**SOP（Standard Operating Procedure）**：標準作業手順書。個々の作業を手順レベルで記載した文書。

| 区分 | 対象 | 粒度 | 例 |
|------|------|------|-----|
| 業務マニュアル | 業務全体 | 概要〜詳細 | 「経理業務マニュアル」「新入社員受入マニュアル」 |
| SOP | 個別作業 | 手順レベル | 「月次決算クロージング手順」「広告レポート作成手順」 |

---

## 従来の課題

| 課題 | 詳細 |
|------|------|
| 通達の書き方がバラバラ | 発信者によってフォーマットが異なり、読みにくい |
| マニュアルが作られない | 忙しくて作成する時間がなく、ノウハウが属人化 |
| 作ったマニュアルが使われない | 内容が古い、場所がわからない、分かりにくい |
| 更新されず陳腐化 | 業務が変わっても文書が更新されない |
| 多言語対応ができない | 外国人従業員への通達・マニュアルが日本語のみ |

## Claudeによる自動化・効率化

### 1. 通達文の即座ドラフト作成
伝達事項のポイントを入力するだけで、フォーマットに沿った通達文を生成。

### 2. マニュアルの構成設計
業務内容を説明すれば、マニュアルの目次構成と各章の骨子を自動設計。

### 3. SOPの手順書作成
作業のポイントを箇条書きで入力すれば、ステップバイステップのSOPを生成。

### 4. 既存マニュアルの改善
既存の文書を読み込ませて、分かりにくい箇所の指摘・改善案を提示。

### 5. 多言語翻訳
日本語の通達・マニュアルを英語等に翻訳。業務用語の正確な訳出も可能。

## プロンプト例（コピペで使えるレベル）

### プロンプト1：人事異動の通達文

```
以下の内容で、人事異動の社内通達文を作成してください。

【異動情報】
- 発令日：2026年4月1日付
- 対象者と異動内容：
  1. 田中太郎：営業部 → 制作部（部長昇格）
  2. 佐藤花子：制作部 → 営業部
  3. 山本一郎：新規採用（管理部配属）

【通達の要件】
- 件名：「人事異動のお知らせ」
- 宛先：全社員
- 発信者：代表取締役 ○○ ○○
- 発令日を明記
- 異動者一覧は表形式（氏名、現所属、新所属、役職）
- 「円滑な引き継ぎにご協力ください」の一文を含む
- 敬体（です・ます調）
- A4で1枚に収まる分量
```

### プロンプト2：制度変更の通知文

```
以下の制度変更について、全社通知文を作成してください。

【変更内容】
- 対象制度：テレワーク制度
- 変更日：2026年5月1日施行
- 主な変更点：
  1. テレワーク実施日数の上限を週2日→週3日に拡大
  2. テレワーク手当を新設（月額5,000円）
  3. テレワーク時の業務報告を日報→週報に簡素化
  4. テレワーク環境の自己チェックシート提出を義務化（年1回）

【通知の要件】
- 変更の背景・目的（働き方改革の推進、従業員満足度向上）
- 変更点を箇条書きで明確に
- 新制度の詳細は社内ポータルの規程集を参照する旨
- 手当の申請方法（管理部へ申請書提出）
- 問い合わせ先：管理部 D
- 敬体、A4で1枚
```

### プロンプト3：注意喚起の通達文

```
以下の内容で、情報セキュリティに関する注意喚起の通達文を作成してください。

【背景】
最近、取引先を装ったフィッシングメールが業界内で増加している。
当社にも不審なメールが複数届いている。

【伝えるべきこと】
1. フィッシングメールの具体的な手口（請求書添付、URL誘導）
2. 不審メールの見分け方（送信元ドメイン、日本語の不自然さ、緊急性を煽る文面）
3. 不審メールを受け取った場合の対応（開かない→管理部に報告→削除）
4. パスワード管理の再確認（使い回し禁止、定期変更）

【通達の要件】
- 件名：「【重要】フィッシングメールへの注意喚起」
- 具体例を含める（実際に届いたメールの特徴を伏せて紹介）
- やるべきこと・やってはいけないことを明確に
- 報告先と連絡方法
- 緊急度が伝わるトーン（ただし過度に不安を煽らない）
```

### プロンプト4：業務マニュアルの構成設計

```
以下の業務について、業務マニュアルの目次構成と各章の概要を設計してください。

【対象業務】新入社員の受入（オンボーディング）
【対象読者】管理部担当者、受入部署の上長

【マニュアルに含めるべき内容】
- 入社前の準備（入社書類の準備、備品の手配、アカウント作成）
- 入社初日のスケジュール（オリエンテーション、挨拶回り）
- 入社1週目のプログラム（社内ルール説明、業務説明、OJT開始）
- 入社1か月目のフォロー（面談、進捗確認）
- 試用期間中の評価と本採用の判断
- 各種届出（社会保険、雇用保険、住民税の特別徴収切替）

【出力要件】
1. マニュアルの目次構成（章→節→項の3階層）
2. 各章の概要（100字程度）
3. チェックリスト形式にすべき箇所の提案
4. テンプレート化すべき文書の一覧（入社案内メール、歓迎メッセージ等）
5. 推奨するページ数の目安
```

### プロンプト5：SOPの作成（広告レポート作成手順）

```
以下の業務について、SOP（標準作業手順書）を作成してください。

【対象業務】月次広告レポートの作成
【対象読者】営業部メンバー（広告運用1年目〜）
【所要時間の目安】1クライアントあたり60分

【作業概要】
1. Google Ads / Meta Ads Manager からデータをエクスポート
2. スプレッドシートのテンプレートにデータを貼り付け
3. 主要KPI（IMP、CTR、CPA、CV数、ROAS）を集計
4. 前月比・目標比を算出
5. 考察コメントを記入（好調/不調の要因分析）
6. Googleスライドのテンプレートに転記
7. 上長レビュー→修正→顧客へ送付

【出力要件】
- ステップバイステップ形式（Step 1, Step 2...）
- 各ステップに：作業内容、使用ツール、所要時間目安、注意点
- スクリーンショットを挿入すべき箇所に[図: ○○画面]のプレースホルダ
- よくあるミスとその防止策
- チェックリスト（提出前の最終確認用）
```

### プロンプト6：既存マニュアルの改善提案

```
以下の既存マニュアル（抜粋）を読み、改善提案をしてください。

【既存マニュアル抜粋】
（ここに既存マニュアルのテキストを貼り付け）

【改善の観点】
1. 分かりやすさ
   - 初めて読む人が迷わず作業できるか
   - 専門用語に説明があるか
   - 手順の粒度は適切か（粗すぎ/細かすぎ）

2. 正確性
   - 手順に抜け漏れはないか
   - 画面や操作が現在のシステムと一致しているか

3. 構成
   - 目次構成は論理的か
   - 必要な情報にすぐアクセスできるか

4. 実用性
   - チェックリストやテンプレートは含まれているか
   - 例外ケースの対応方法は記載されているか

【出力形式】
- 改善点一覧（重要度：高/中/低、該当箇所、問題点、改善案）
- 改善後のサンプル（特に問題のある箇所を書き直した例）
```

### プロンプト7：マニュアル更新の差分管理

```
以下の業務マニュアルの更新作業を支援してください。

【更新理由】
使用ツールが変更になった（Chatwork → Slack への移行）

【対象マニュアル】
（ここにマニュアルのテキストを貼り付け）

【出力要件】
1. 「Chatwork」に関連する記載箇所を全て洗い出し
2. 各箇所の修正案（Slackの操作に置き換え）
3. ツール移行に伴い新たに追加すべき手順
4. 削除すべき手順
5. 新旧対照表（変更前 → 変更後）
6. 更新履歴への記載文案
```

## 注意点

1. **通達の法的効力** — 社内通達は、就業規則や労働協約に基づく業務命令としての効力を持つ場合がある。人事異動の発令通知は労働契約上の根拠が必要であり、就業規則に異動命令の根拠規定があることを確認すること。

2. **個人情報への配慮** — 人事異動の通達に個人の事情（病気、介護等）を記載しないこと。また、懲戒処分を社内通達する場合は、対象者のプライバシーに十分配慮する。

3. **マニュアルの著作権** — 他社のマニュアルをそのままコピーすることは著作権侵害となる。Claudeで生成したマニュアルも、他社の公開文書を参考にしている可能性があるため、自社の業務に合わせて十分にカスタマイズすること。

4. **バージョン管理の徹底** — マニュアルは必ず版数・改定日・改定内容を管理する。古いバージョンが流通しないよう、「現行版はここにある」という一元的な保管場所を定めること。

5. **読者目線のテスト** — マニュアルを作成したら、実際にそのマニュアルだけを見て作業できるか、対象読者にテストしてもらう。Claudeに「この手順書を初見で読んだ人が迷うポイントを指摘して」と依頼するのも有効。

6. **多言語対応** — 外国人従業員がいる場合、重要な通達やマニュアルは多言語化を検討する。Claudeは多言語翻訳が可能だが、業務固有の用語は必ず人間がレビューすること。

## まとめ

- 社内通達は **統一フォーマット** を定め、誰が書いても品質が均一になる仕組みを作る
- 業務マニュアルは **「初めての人が迷わず作業できる」** を基準に、チェックリストやテンプレートを活用する
- SOPは **手順レベルの粒度** で記載し、所要時間・注意点・例外ケースまで網羅する
- **バージョン管理と定期見直し** で、文書の陳腐化を防ぐ
- Claudeを活用すれば、**通達文のドラフト・マニュアルの構成設計・SOP作成・既存文書の改善** を大幅に効率化できる
- 完成した文書は **対象読者にテスト** してもらい、実用性を検証してから正式運用すること
