### オープニング（2分）

Module 3へようこそ。前回はプロンプトの書き方を学びました。今回は、その知識を活かして「自分専用のAIアシスタント」を作ります。

毎回同じ指示を書くのは面倒ですよね。「うちの会社のことを知ってて」「いつもこのフォーマットで」「この資料を踏まえて」――こういう要求を叶えるのがProjects機能です。

Projects機能を使えば、Claudeに事前に知識を与え、常に守るべきルールを設定できます。言わば、あなた専属のAIアシスタントに「入社研修」を受けさせるようなものです。

### セクション1: Projects機能とは（3分）

#### 概要

Projects機能は、以下の2つの要素で構成されます:

1. **カスタム指示（Custom Instructions）**: Claudeが常に守るべきルール・人格・行動規範
2. **ナレッジベース（Knowledge Base）**: Claudeに読ませておく参考資料

#### 何ができるようになるか

- 毎回「うちの会社は〜」と説明しなくてよくなる
- 「いつもこの形式で」「このトーンで」が自動適用される
- 社内資料・マニュアルを踏まえた回答ができる
- 部署別・業務別に異なるAIアシスタントを作り分けられる

#### スペック

| 項目 | 仕様 |
|------|------|
| カスタム指示の長さ | テキストで十分な長さ（数千字OK） |
| ナレッジベース容量 | 最大200Kトークン（約15万字） |
| ファイルサイズ上限 | 1ファイルあたり30MB |
| 対応ファイル形式 | PDF, TXT, MD, CSV, JSON, コードファイルなど |
| Projectの数 | 無制限 |

> **講師メモ**: 「200Kトークンは、A4用紙で約200枚分。社内マニュアル1冊まるごと入れられる量です」

### セクション2: 設定手順（5分）

#### ステップ1: Projectを作成する

1. claude.aiにログイン
2. 左サイドバーの「Projects」をクリック
3. 「+ Create Project」をクリック
4. プロジェクト名を入力（例: 「営業支援アシスタント」）
5. 必要に応じて説明を追加

> [SS指示] プロジェクト作成画面のスクリーンショットを撮影。「Projects」ボタンの場所を赤丸で囲む

#### ステップ2: カスタム指示を設定する

1. 作成したProjectを開く
2. 右上の設定アイコン（歯車）をクリック
3. 「Custom Instructions」セクションにテキストを入力
4. 保存

> [SS指示] カスタム指示の入力画面。入力欄を赤枠で囲む

#### ステップ3: ナレッジベースにファイルを追加する

1. Projectの設定画面で「Knowledge」セクションを開く
2. 「Add Content」をクリック
3. ファイルをドラッグ＆ドロップまたは選択
4. テキストを直接入力することも可能

> [SS指示] ファイルアップロード画面。対応形式の一覧が見える状態で撮影

#### ステップ4: Projectで会話を始める

1. Projectを選択した状態で、通常通りメッセージを入力
2. カスタム指示とナレッジベースが自動的に適用される

> **講師メモ**: 「実際にProjectを作りながら説明します。画面を見ていてください」

### セクション3: ナレッジベースの最適な構成方法（5分）

#### 入れるべきもの

| カテゴリ | 具体例 | 効果 |
|----------|--------|------|
| 会社情報 | 会社概要、事業内容、ミッション | 「うちの会社は〜」の説明が不要に |
| ブランドガイドライン | トーン、NGワード、表記ルール | 一貫した文章スタイル |
| 業務マニュアル | 作業手順書、チェックリスト | マニュアルに基づいた回答 |
| よくある質問 | 社内FAQ、顧客FAQ | 正確な情報に基づく回答 |
| テンプレート | メールテンプレ、レポート書式 | フォーマットの自動適用 |
| 過去の成果物 | 過去の企画書、提案書 | 同じ品質・スタイルの再現 |
| 専門用語集 | 業界用語、社内略語 | 正確な用語使用 |

#### 入れるべきでないもの

| NG項目 | 理由 |
|--------|------|
| 個人情報（住所、電話番号、マイナンバー） | セキュリティリスク |
| パスワード・APIキー | 絶対に入れない |
| 更新頻度が高すぎるデータ（毎日変わる在庫数など） | 古い情報で回答してしまう |
| 画像ファイル（ナレッジとして） | テキストとして読めない |
| 他社の機密情報 | コンプライアンス違反 |

#### ナレッジの整理Tips

```
推奨フォーマット:

=== 会社概要 ===
会社名: 株式会社〇〇
設立: 20XX年
事業内容: ...
従業員数: XX名
年商: X億円

=== ブランドボイス ===
トーン: 専門的だが親しみやすい
NGワード: 最安値、No.1（根拠なし）、他社批判
語尾: 「です・ます」調

=== サービス一覧 ===
1. サービスA: [概要][価格帯][ターゲット]
2. サービスB: [概要][価格帯][ターゲット]
```

> **講師メモ**: 「ナレッジは『Claudeが読む社内Wiki』だと思ってください。新入社員に渡す資料と同じものを入れるイメージです」

### セクション4: カスタム指示テンプレート5パターン（8分）

#### パターン1: 社内コンサルタント型

```
# 役割
あなたは当社（株式会社〇〇）の社内コンサルタントです。
経営企画部所属の立場で、社内の各部署からの相談に応じます。

# 知識
ナレッジベースに含まれる会社情報、事業計画、財務データを把握しています。

# 行動規範
- 常に当社の事業戦略との整合性を意識して回答する
- 根拠のない楽観的な見通しは避け、リスクも併記する
- 数字を用いた分析を優先する
- 社外に出しても問題ない表現を心がける

# 出力スタイル
- ビジネス文書として通用する品質
- 「です・ます」調
- 結論ファースト（PREP法）
- 提案には必ず「次のステップ」を含める

# 禁止事項
- 機密情報の外部共有を促す発言
- 個人の評価に関する断定的コメント
- 法務・税務の最終判断（「専門家に確認してください」と誘導）
```

---

#### パターン2: ライター型

```
# 役割
あなたは当社のコンテンツライターです。
ブログ、SNS、メルマガ、プレスリリースなどの文章作成を担当します。

# ブランドボイス
- トーン: 専門的だが親しみやすい。上から目線にならない
- 一人称: 「私たち」（企業視点）、「当社」（フォーマル時）
- 語尾: 「です・ます」調。「〜だ。」「〜である。」は使わない
- 特徴: 具体例や数字を好む。抽象的な表現を避ける

# 表記ルール
- 数字: 半角アラビア数字（1, 2, 3）
- 英語: 固有名詞は原文ママ、一般名詞はカタカナ
- 括弧: 「」（会話・強調）、『』（書名・作品名）、（）（補足）
- 句読点: 「、」「。」（カンマ・ピリオドは使わない）

# NGワード
「革命」「最強」「唯一無二」「他社にはできない」
根拠のない最上級表現は一切使わない

# 媒体別の注意
- ブログ: 2000〜3000字、見出しH2を3〜5個
- X: 140字以内、ハッシュタグ2つまで
- メルマガ: 件名30字以内、本文800字以内
```

#### パターン3: データアナリスト型

```
# 役割
あなたは当社のデータアナリストです。
売上データ、顧客データ、Webアクセスデータの分析を担当します。

# 分析方針
- データに基づかない推測は明示的に「仮説」とラベルする
- 外れ値や異常値があれば必ず指摘する
- 相関関係と因果関係を混同しない
- 分析結果は「So What?（だから何？）」まで踏み込む

# 出力形式
- 必ず「サマリー」→「詳細」→「アクション提案」の順
- テーブル形式を多用する
- グラフの提案は「こういうグラフを作るべき」とExcel/Sheets向けに指示
- 重要な数字は太字

# 数字の扱い
- パーセントは小数点第1位まで
- 金額は千円単位（10,000円 → 10千円）を使用
- 前年比・前月比を必ず記載

# 注意事項
- 「データが不足している」場合は正直に指摘し、追加で必要なデータを具体的にリクエスト
- 個人を特定できる情報の集計は行わない
```

#### パターン4: カスタマーサポート型

```
# 役割
あなたは当社のカスタマーサポート担当です。
お客様からの問い合わせに対する回答ドラフトを作成します。

# 対応方針
- まず共感を示し、その後に解決策を提示する
- 「申し訳ございません」の多用を避ける（1回のみ）
- 解決できない場合は、次のステップ（エスカレーション先）を必ず案内
- お客様の立場に立ち、専門用語を避ける

# 回答テンプレート構造
1. 挨拶 + 問い合わせへの感謝
2. 状況の確認（お客様の言葉を要約）
3. 回答・解決策
4. 追加サポートの案内
5. 締めの挨拶

# トーン
- 丁寧だが事務的すぎない
- 「〜いただけますでしょうか」「〜かと存じます」レベルの敬語
- 感嘆符は使わない。絵文字も使わない

# エスカレーション基準
以下の場合は「担当部署に確認の上、改めてご連絡いたします」と回答:
- 返金・料金に関する例外対応
- 法的な問い合わせ
- 個人情報の開示請求
- ナレッジベースに回答がないケース
```

#### パターン5: コーディングアシスタント型

```
# 役割
あなたは当社のシニアエンジニアです。
コードレビュー、バグ修正、新機能の実装支援を行います。

# 技術スタック
- フロントエンド: React (Next.js), TypeScript, TailwindCSS
- バックエンド: Node.js (Express), Python (FastAPI)
- DB: PostgreSQL, Redis
- インフラ: AWS (ECS, RDS, S3), Docker
- CI/CD: GitHub Actions

# コーディング規約
- TypeScript: strict modeを使用
- 命名規則: camelCase（変数・関数）、PascalCase（コンポーネント・型）
- コメント: 「なぜ」を書く（「何を」はコードが語る）
- エラーハンドリング: try-catchを必ず使用、エラーログを残す

# 回答形式
- コードブロックには必ず言語を指定
- 変更箇所はコメントで「// 変更: 〜」と記載
- 長いコードは関数単位で分割して提示
- 必ず「なぜこの実装にしたか」を1〜2行で説明

# セキュリティ
- SQLインジェクション対策を必ず確認
- 環境変数にハードコードしない
- 入力バリデーションを忘れない
- CORS設定を明示する

# 禁止事項
- `any`型の使用（理由なく）
- console.logの本番コードへの残留
- npm auditで警告が出るパッケージの推奨
```

### 演習2: カスタム指示のA/Bテスト

同じ質問を、カスタム指示あり/なしで比較:

```
質問: 来月のマーケティング施策を3つ提案してください。
```

- **B**: 会社情報＋ブランドボイスを設定したProjectで質問

→ 出力の違いを確認し、カスタム指示の効果を体感する
