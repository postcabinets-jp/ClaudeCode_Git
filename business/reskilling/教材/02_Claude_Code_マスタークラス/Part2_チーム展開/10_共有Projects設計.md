### [0:00〜2:00] オープニング

Module 07でTeamプランを契約し、08でNotionを繋ぎ、09でSlackを繋ぎました。インフラは整いました。

でも、まだ足りないものがあります。

Claudeに「うちの会社のルール」「うちの部署のやり方」「このプロジェクトの背景」を教えていない。毎回、毎人、同じ説明をClaudeにしている状態です。

「見積書のフォーマットはこうで」

部署の知恵をAIに集約する。今日はその設計と運用の話です。

### [2:00〜5:00] 共有Projectsとは

まず基本的な仕組みを確認します。

**Projectの構成要素**:

| 要素 | 内容 | 例 |
|------|------|-----|
| プロジェクト名 | 識別名 | 「マーケ部ガイドライン」 |
| 説明文 | 目的の簡潔な記述 | 「マーケティング部の業務でClaudeを使う際の共通設定」 |
| カスタム指示 | Claudeへの指示文 | 「あなたはPOST CABINETSのマーケ担当です。ブランドガイドラインに従って...」 |
| ナレッジファイル | 参照資料のアップロード | ブランドガイドライン.pdf、過去の成功事例.md |
| 可視性 | 公開範囲 | Private / Team |

**個人Project vs 共有Project**:

| | 個人Project | 共有Project |
|---|-----------|-----------|
| 可視性 | 自分だけ | チーム全員（or 指定メンバー） |
| 作成者 | 誰でも | Admin推奨 |
| 編集権限 | 作成者 | Admin / Editor |
| ユースケース | 個人の作業効率化 | チーム標準の確立 |

共有Projectで会話を始めると、カスタム指示とナレッジが自動で適用されます。メンバーは何も設定せずに、チームの文脈を持ったClaudeと会話できる。

### [5:00〜12:00] 共有Projects設計パターン

3層構造で設計します。全社 → 部署 → プロジェクト。

#### 第1層: 全社共通Project

全社員が使う、会社としての共通ルール。

**Project名: POST CABINETS 全社共通**

カスタム指示の例:
```
あなたはPOST CABINETS（株式会社ポストキャビネッツ）の業務アシスタントです。

## 会社概要
- 事業: Webマーケティング・SNSマーケティング支援
- 設立: 20XX年
- 所在地: 大阪府
- 従業員: XX名

## コミュニケーションスタイル
- 敬語ベース、ただし堅すぎない
- 専門用語は必要に応じて使うが、社外向けには平易に言い換える
- 結論ファースト

## ブランドガイドライン
- ブランドカラー: #XXXXXX
- トーン: 信頼感がありつつ親しみやすい
- NGワード: 「絶対」「必ず」「100%」（断定的な表現は避ける）

## セキュリティルール
- 顧客の個人情報（氏名・電話番号・メールアドレス）は入力しない
- 契約金額の具体数値は入力しない
- 社内の人事情報は入力しない
- 不明な場合は「この情報をClaudeに入力してよいか上長に確認してください」と促す
```

ナレッジファイル:
- `brand-guideline.pdf` — ブランドガイドライン
- `security-policy.md` — AI利用セキュリティポリシー
- `company-overview.md` — 会社概要

### [12:00〜15:00] カスタム指示の設計方法（チーム版）

個人で使うカスタム指示と、チームで使うカスタム指示は設計思想が異なります。

**個人のカスタム指示**: 「自分の好み」を反映
```
出力はmarkdownで。箇条書きを多用して。日本語で回答して。
```

**チームのカスタム指示**: 「組織のルール」を反映
```
POST CABINETSのブランドガイドラインに従って回答してください。
顧客の個人情報は入力しないでください。
見積書のフォーマットはナレッジファイルを参照してください。
```

**設計の5原則**:

1. **明確にする**: 曖昧な指示は人によって解釈が変わる
   - NG: 「丁寧に回答して」
   - OK: 「敬語を使い、結論を最初に述べ、根拠を箇条書きで示してください」

2. **禁止事項を明示する**: やってはいけないことを先に書く
   - 「個人情報は扱わない」「断定的な表現は使わない」「法的アドバイスはしない」

3. **具体例を含める**: 良い例・悪い例を併記
   ```
   ## メール文面のトーン
   OK: 「お世話になっております。ご依頼の件、以下の通りご提案いたします。」
   NG: 「了解です！すぐやりますね〜」
   ```

4. **更新しやすい構造にする**: セクション分けを明確に
   ```
   ## 会社概要
   ## コミュニケーションスタイル
   ## セキュリティルール
   ## 業務別ルール
   ```

5. **長すぎない**: カスタム指示は2,000文字以内が目安。詳細はナレッジファイルに

### [15:00〜18:00] ナレッジベースの運用ルール

ナレッジファイルは、アップロードしたら終わりではありません。運用ルールが必要です。

**誰が更新するか**

| レイヤー | 更新責任者 | 頻度 |
|---------|-----------|------|
| 全社共通 | 管理者（Admin） | 四半期に1回 |
| 部署別 | 部署リーダー | 月1回 |
| プロジェクト別 | PM | 随時（マイルストーンごと） |

**いつ見直すか**

トリガーベースで更新タイミングを決めます。

| トリガー | アクション |
|---------|----------|
| 社内ルール変更 | 全社共通Projectを更新 |
| 新しい案件開始 | プロジェクト別Projectを新規作成 |
| 案件終了 | プロジェクト別Projectをアーカイブ + 成功事例を部署Projectに反映 |
| 新メンバー入社 | 部署Projectの内容で不足がないか確認 |
| 四半期ごと | 全Projectの棚卸し（不要なもの削除、内容の鮮度確認） |

**ナレッジファイルの命名規則**

```
[カテゴリ]-[内容]-[バージョン].md

例:
brand-guideline-v2.md
sales-proposal-template-v1.md
client-x-requirements-v3.md
```

**ファイル形式の推奨**

| 形式 | 推奨度 | 理由 |
|------|--------|------|
| .md（Markdown） | 最推奨 | トークン効率が最も良い、構造化しやすい |
| .txt | 推奨 | シンプル、軽量 |
| .pdf | 可 | レイアウトが重要な場合 |
| .docx | 可 | 既存資料の流用 |
| .xlsx | 可 | 表データ（価格表等） |
| .pptx | 非推奨 | トークン効率が悪い、テキスト化して.mdにする方がよい |

### [18:00〜21:00] 「AIが使える状態を維持する」メンテナンスガイド

最もよくある失敗は「最初は設定したけど、半年後には情報が古くなって使われなくなった」です。

これを防ぐための仕組みを3つ紹介します。

**仕組み1: 月次レビューを予定に入れる**

Googleカレンダーに「Claude Project棚卸し」を毎月1日にセット。15分で以下を確認:
- カスタム指示の内容は現状と合っているか
- ナレッジファイルに古い情報はないか
- 新しく追加すべき情報はないか

**仕組み2: フィードバックチャンネルを設ける**

Slackに `#claude-feedback` チャンネルを作成。メンバーが以下を投稿:
- 「Claudeがこういう回答をしたけど、うちのルールと違う」→ カスタム指示の修正
- 「この情報がClaudeに入っていたら便利なのに」→ ナレッジの追加
- 「このProjectがすごく使いやすい」→ 成功パターンの横展開

**仕組み3: オーナーシップを明確にする**

各Projectに「オーナー」を1名割り当て。Project説明文に明記:
```
オーナー: 田中太郎（マーケ部リーダー）
最終更新: 2026-03-29
次回レビュー予定: 2026-04-30
```

オーナーがいないProjectは腐ります。必ず誰かの名前を入れてください。

### [24:00〜25:00] まとめ

1. **3層構造**: 全社 → 部署 → プロジェクトの階層で設計
2. **カスタム指示の5原則**: 明確・禁止事項・具体例・構造化・簡潔
3. **ナレッジはMarkdownが最強**: トークン効率と構造化のバランス
4. **メンテナンスが命**: 月次レビュー、フィードバックチャンネル、オーナー制
5. **設計してから作る**: いきなりProjectを作らない。体系図を先に描く

次のModule 11では、これらの投資に対する効果測定。KPI設計とROI計算を具体的な数字で見ていきます。
