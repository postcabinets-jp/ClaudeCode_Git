# /think — 着手前プロトコル

これから取り組むタスク「$ARGUMENTS」について、コードや成果物を書き始める**前に**以下を整理してください。

## 1. 真の目的の言語化

このタスクの裏にある「本当に欲しいもの」を1文で書く。表面の指示そのままを繰り返さない。

例：
- 表面: 「ログイン画面作って」
- 真の目的: 「サービスの初回体験を阻害せず、かつセキュリティを最低限担保したい」

## 2. 不明点の洗い出し

不明点を**1〜3個に絞って**列挙する。質問が10個出るなら、それは設計が無いシグナルなのでアプローチを変える。
質問は「Yes/Noで答えられる形」「2〜3択」のどちらかで聞く。自由記述で聞くのは最終手段。

## 3. アプローチの提示

確認が不要なら、これからやるアプローチを**3〜5行**で書く。
判断が分かれる点（ライブラリ選択・構造）があれば候補を2つ並べて推奨を示す。

## 4. 完了条件

「何ができたら完了か」を1〜2行で書く。曖昧なら依頼者に確認する。
