# /tech-debt — 技術的負債を可視化して計画化

対象：`$ARGUMENTS`（パス指定 or `all`）

## 診断項目

### 1. コード品質
- LOC / ファイル
- Cyclomatic complexity（複雑度10超は要注意）
- 重複コード（jscpd, similarity-py）
- TODO / FIXME / XXX コメント数

### 2. テスト
- カバレッジ（line / branch）
- テスト/コード比率
- 1ヶ月以上更新ないテストファイル

### 3. 依存
- deprecated パッケージ
- 2 major version 以上遅れている依存
- 脆弱性（HIGH/CRITICAL）

### 4. ドキュメント
- README が3ヶ月以上更新なし
- 関数/クラスにコメントない（大規模プロジェクト）
- 死んだリンク

### 5. アーキ
- 循環依存
- god class（500行超）
- god function（100行超）

## 出力フォーマット

```
## 負債スコア
- コード: X/100
- テスト: X/100
- 依存: X/100
- ドキュメント: X/100
- アーキ: X/100
合計: XX/500

## ホットスポット（上位5件）
1. src/foo.ts (LOC 800, 複雑度 25, テストなし)
2. ...

## 改善プラン（優先度順）

### 高優先（次のスプリント）
- [タスク] / 工数 / 効果

### 中優先（1ヶ月以内）
- ...

### 低優先（時間あれば）
- ...

## 推定削減効果
- 改善後の負債スコア見込み
- 開発速度への影響
```

## 禁止
- 「全部リファクタ」提案（コスパ無視）
- 計測なしでの「ここが汚い」
- 流行りに乗せ替え提案（Webpack→Vite を理由なく等）
