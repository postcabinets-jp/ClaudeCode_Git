複数機能を「チーム組んで並行実装」する依頼で、単一コードベース（Flutter等）にそのままN体のエージェントを放つと共有ファイル（DBスキーマ/リポジトリ/provider/モデル）でほぼ確実に衝突する。

**勝ちパターン（iron_log v5で7機能を並行投入し衝突ゼロ・型不整合ゼロを達成）:**
1. **基盤を親が先に1本化**: DBスキーマ・リポジトリの全新規メソッド・provider・モデル・共有widget（チャート等）を親自身が書き、`analyze`＋`build`が通る状態にする。新画面は「コンパイル可能なスタブ」を置いて契約（クラス名・コンストラクタ）を固定。
2. **1エージェント=1ファイル所有**: 各機能エージェントに「あなたが編集してよいのはこの1ファイルのみ。共有ファイルは触るな。`build`/`run`/`analyze`も実行するな（並行編集中だから）」と厳命。新規widgetが要るなら衝突しない新ファイル名を割当。
3. **基盤APIの完全な仕様書を各プロンプトに添付**: メソッドの厳密なシグネチャ・provider名・モデルのフィールド・テーマ色・デザイン禁則。これで「足りないから共有ファイルを勝手に拡張」が起きない。
4. **統合は親が一手に**: 全エージェント完了後に親が`analyze`→軽微lint修正→`build`→実機目視→quality-reviewer。

**Why:** 7体並行でも残ったのは軽微lint5件のみ。共有ファイルへの書き込みを物理的に1人（親）に集約したから衝突しなかった。
**How to apply:** 「並行で全部実装」系の大依頼。まず破壊的変更を含む共有層を自分で固め、独立UIだけ並列に出す。worktree分割よりこの方式が単一コードベースでは確実（マージ地獄を回避）。IndexedStack等でタブ常駐させると別タブのproviderが古キャッシュ化するので、更新元で関連providerを全部invalidateするのを忘れない。
