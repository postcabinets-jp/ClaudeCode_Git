朝会(SUM)・予定確認では、カレンダーを**目視で要約しない**。`fetch_calendars.py [YYYY-MM-DD]` の正規化出力を唯一の事実ソースにする。

**実行**: `/Users/apple/note/_system/.venv/bin/python ~/.note-system/scripts/fetch_calendars.py 2026-06-15`
（7口を Calendar API 直叩きで正規化Markdown出力。控室/府議団は「参考」フラグ付き＝全議員配信でNobu確定ではない）

**やってはいけないこと（2026-06-15の実際のミス）**:
- ❌ mtg_prep が04:00に生成した「今日の予定」表をそのまま信じて喋る → prepのカレンダーはズレる（無人MCPがEVENTS=0を返す既知問題のため）
- ❌ MCPで複数口の生JSONを取って、目視でざっと要約する → 今日の幹事会13:00・議員団総会13:30を読み落とし「全部明日」と誤った一般化をした

**1点だけ残る目視チェック**: スクリプトは終日イベントの翌日混入が弱い（tmaxが翌朝6時まで＝翌日の終日予定が今日側に出る）。「終日」表示のイベントだけ日付を1点確認する。例: 2026-06-15出力に「終日 本会議（採決・閉会）」が出たが実際は6/16。

**Why:** 2026-06-15、prep鵜呑み→生JSON目視と2連続でカレンダーを外し、Nobuに「毎回ちょいちょい違うと面倒。きっちりやれるんか分からん」と指摘された。根本原因は「一次データを機械的に全件確認せず、思いつきで要約した」こと。
**How to apply:** カレンダー・予定が絡む発話の前に必ずスクリプトを回す。俺(Claude)がJSONを読んで要約する工程を挟まない＝読み落としが構造的に起きなくなる。これはカレンダーに限らず「データ仕事は全件確認/スクリプトを挟む、目視要約しない」の一例。
