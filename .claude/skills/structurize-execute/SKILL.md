# Structurize & Execute（構造化→即実行）

nobuの生の発言を噛み砕いて構造化し、実現スピードを最大化するスキル。

## フロー

### Step 1: インプット受け取り

nobuからの入力を受け取る。形式は問わない:
- 音声書き起こし（文脈が飛ぶ、主語が省略される、話が戻るのは普通）
- チャットメッセージ（断片的でOK）
- 箇条書きメモ
- 「〜したい」「〜どうかな」的なぼやき

### Step 2: 構造化（即座に）

以下のフォーマットで整理して提示する:

```
## 🎯 ゴール
[nobuが本当にやりたいこと。1文で。]

## 📌 前提・制約
- [関係する背景やリソースの制約]
- [期限があれば]
- [技術的な前提があれば]

## 🔧 ステップ
1. **[最初にやること]** ← いまここ
2. [次にやること]
3. [その次]
（最大5ステップ。それ以上は分割する）

## ⚡ 前提の仮定（曖昧だった部分）
- [判断が必要だった点と、こちらで仮置きした内容]
```

### Step 3: 確認→即実行

構造化を提示したら:

> 「これで進める？修正あれば言って。なければStep 1から始める。」

と聞く。

- **OKが出たら**: Step 1を即座に実行開始する
- **修正が入ったら**: 構造を修正して再提示（ただし1往復まで。2回目は修正を反映して実行開始）

## ルール

1. **質問は最大1回**。曖昧な点は合理的な仮定を置いて明記する。「わからないから聞く」は最小限にする
2. **行動バイアス**。完璧な計画より、動き出せる計画を優先する
3. **簡潔に**。構造化の出力は読むのに30秒以内で済む分量にする
4. **実行可能なものは即実行**。コード変更・ファイル作成・Notion更新などは確認後すぐ手を動かす
5. **外部ツール連携が必要な場合**: アクションを明示して承認を得てから実行する（Notion書き込み、Discord投稿など）
6. **言語は日本語**。技術用語は英語のままでOK
7. **トーンはカジュアル・ダイレクト**。敬語は最小限。「〜です・ます」より「〜する・〜だ」寄り

## 実行時の判断基準

| 状況 | 対応 |
|------|------|
| コード/ファイルの変更 | 確認後、即座に実装開始 |
| Notionへの記録 | 構造化した内容をNotionに書く提案→承認後実行 |
| 調査が必要 | 調査を最小限に絞り、結果をステップに反映 |
| 複数プロジェクトにまたがる | プロジェクトごとにステップを分離して提示 |
| スコープが大きすぎる | 「今日やる分」と「後回し分」に分割する |

## 入力例と出力例

### 入力
> 「LINE Botでユーザーが質問したらAIが答える仕組み作りたいんよな。COPAINのやつ。まずはテキストだけでええわ。Supabaseにログも残したい。」

### 出力

## 🎯 ゴール
COPAIN LINE BotにAIチャット応答機能を追加する（テキストのみ）

## 📌 前提・制約
- 既存のLINE Bot基盤を使う
- AI応答はClaude API（既にプロジェクトで使用中）
- 会話ログはSupabaseに保存

## 🔧 ステップ
1. **LINE Webhook → Claude API呼び出しの基本フロー実装** ← いまここ
2. Supabaseにメッセージログテーブル作成・保存処理追加
3. 動作確認（LINE実機テスト）

## ⚡ 前提の仮定
- モデルはClaude 3.5 Sonnetを想定（コスト・速度バランス）
- システムプロンプトは汎用的なものを仮置き（後で調整）
