実装アプローチ比較

どの構成で自動化パイプラインを組むか選んでください

A

launchd + Node.js スクリプト（推奨）

既存の autonomous-loop.mjs / notion-pulse.mjs パターンを踏襲。2つのlaunchd jobで完結。

✅ メリット
  • 既存インフラそのまま使える
  • 追加依存なし
  • Mac常駐・確実に動く
  • コードシンプル
⚠️ デメリット
  • Macが起動していないと動かない
  • スリープ中は実行されない
job1（毎朝7時）: Claude API で下書き生成 → Notionに保存 → Discord通知
job2（15分ごと）: Notion「投稿OK」をポーリング → Playwright でLinkedIn投稿
B

Claude Code スケジュール（RemoteTrigger）

Claude Codeのスケジュール機能でエージェントを起動。コード最小限。

✅ メリット
  • スクリプト実装量が少ない
  • Claude Code管理画面で制御可能
⚠️ デメリット
  • Claude Codeセッション依存
  • トークンコスト増
  • 長期安定性に懸念
/schedule で毎朝・毎15分のエージェントを登録。Notion読み書き・Playwright操作をClaudeが直接実行。
C

A + Webhook（ハイブリッド）

launchd基盤にNotionのWebhookを追加。「投稿OK」と同時にリアルタイム投稿。

✅ メリット
  • チェックと同時に即時投稿
  • ポーリング不要
⚠️ デメリット
  • Webhook受信サーバーが必要
  • 構成が複雑になる
  • Notion Webhookは有料プランのみ
Notion → Webhook → ローカルサーバー → Playwright投稿。リアルタイムだが複雑。
💡 おすすめ: A
既存の autonomous-loop.mjs や notion-pulse.mjs と同じパターンなので、コードを書く量が最小限。launchd で安定稼働。Cは現時点では過剰設計。
